## [1.5.1] - 2026-07-18

### Fixed
- **Listing pagination silently lost one object per page boundary** — `NextMarker` was set to the first key of the NEXT page while marker consumption follows S3 semantics (resume strictly AFTER the marker), so any client paging with the returned marker skipped exactly one object per 1,000 (console listings, S3 ListObjects, SOSAPI, search). At backup-target scale this made verification/repair tools (Veeam, Duplicati) see existing files as missing — ~500 invisible objects on a 500K-file listing. Fixed in all three paginated paths (`ListObjects`, `ListObjectsDelimited` — including resuming after a common-prefix marker by skipping the whole prefix group — and `searchObjectsByScan`); found by an end-to-end torture test (1,090 files, 5 hard kills, worker conversions, KEK rotation, churn — zero losses after the fix), covered by pagination round-trip regression tests at page sizes 1-100. (`internal/metadata/pebble_objects.go`, `internal/metadata/pebble_pagination_test.go`)

### Changed
- **Dead-code sweep across backend and frontend** — removed code with zero production callers, verified independently with `deadcode` (whole-program reachability), `staticcheck` (U1000) and per-symbol `git grep`, plus manual confirmation of the live equivalent for every removal. Highlights:
  - **Parallel implementations that were never wired** (each has a live counterpart that stays): `internal/object/lock.go` + `retention.go` (live object lock/retention = the `Manager` interface methods in `manager.go`), `internal/middleware/ratelimit.go` (live = `authManager.CheckRateLimit` + dynamic settings), `internal/cluster/rate_limiter.go` (the server constructed it but never called it — including a startup log line that falsely claimed cluster APIs were rate-limited; cluster port protection is TLS + HMAC), `internal/cluster/metrics.go`, `middleware.SignRequest` (live signer = `cluster/proxy.go`), the CSR pair `GenerateKeyAndCSR`/`SignCSR` (live cert issuance = `GenerateNodeCert`), the s3compat cluster-proxy auth shim, the unregistered `prometheusCollector` (live `/metrics` = the manager's own registry), and `pkg/encryption`'s pre-envelope `EncryptionService`/`KeyManager`.
  - **Dead helpers and config variants**: unused CORS/logging config constructors, tracing context helpers, `S3AuthHelper`'s V2/V4 parsing chain (**`GetS3Action`/`GetResourceARN` deliberately kept** as IAM primitives), unused bucket validation/permission helpers, assorted single functions (`getDiskUsage`, `queryBucketsFromNode` server duplicate, `calculateMultipartHash`, `httpHeadersFromObject`, etc.) and the `main.go` assignment that fed `trusted_proxies` into the dead middleware (the live consumer `getClientIP` reads the config directly).
  - **Deliberately kept**: `Stop()` lifecycle methods on sync managers, test seams (`MigrateTo`, `NewManagerWithS3Factory`, `detectMode`, s3compat presigned URL generation — it round-trips the presigned *validation* tests), and `bandwidth.Manager.Remove` (documented optional housekeeping).
  - **Frontend (knip)**: removed 8 unused files (`PageHeader`, `PermissionsEditor`, `useLockedUsers`, 3 stale test mocks, `MetricBarChart`, `MetricPieChart`) and 2 unused dependencies (`i18next-browser-languagedetector`, `@tailwindcss/vite`).

### Added
- **Pebble durability on hard kill (bounded loss + self-healing boot)** — hot-path metadata commits were `NoSync`, so a hard kill (power loss, OOM, `kill -9`) could lose the last seconds of Pebble writes: a just-PUT object kept its data file + sidecar but vanished from listings, and a lost delete tombstone resurrected a ghost entry. Three-layer fix, none of which meaningfully costs throughput:
  - **Periodic WAL fsync (everysec)**: a background loop in `PebbleStore` fsyncs the WAL once per second whenever there are unsynced writes (one fsync/second flat, regardless of write volume — Pebble's group commit makes everything before it durable), bounding hard-kill metadata loss to ≤1s. `WALBytesPerSync` smooths background flushing so the periodic fsync stays cheap. Interval configurable via `PebbleOptions.WALSyncInterval`.
  - **Sync on destructive, low-frequency operations**: object/version deletes, bucket create/update/delete, multipart complete/abort and raw-KV deletes now commit with `pebble.Sync` — a lost delete tombstone would leave a ghost listing entry pointing at a removed file (unservable), and a completed multipart upload is the durability point the client paid for. Hot paths (object puts, parts, metrics) stay `NoSync` under the 1s loop.
  - **Unclean-shutdown reconciliation**: `Close()` writes a `CLEAN_SHUTDOWN` sentinel that the next open consumes; when a pre-existing store opens without it, the server reconciles the metadata store against the on-disk object tree in the background while serving traffic (reads already work via the sidecar fallback). Repairs are non-destructive: entries lost in the crash window are rebuilt from their sidecars (original timestamps preserved), but metadata entries and sidecars are never removed merely because an object path is absent. Bucket stats are recalculated after changes. (`internal/metadata/pebble_store.go`, `internal/recovery/reconcile.go`, `internal/server/unclean_shutdown_reconcile.go`)

## [1.5.0] - 2026-07-10

### Added
- **Envelope encryption with the KEK in the database (always-on encryption)** — server-side encryption no longer depends on a fixed key in `config.yaml` and is now **always active**, matching AWS S3's SSE-S3-by-default model. Highlights:
  - **KEK in the DB**: new `encryption_keys` SQLite table (migration 16) and `internal/kek` package. On startup the KEK is resolved in order: existing DB value → seeded from `config.yaml` `storage.encryption_key` (existing deployments keep decrypting everything; keep the config file as a key backup) → freshly generated 32-byte key persisted in the DB (deployments started with only `--data-dir` self-provision, no config needed). The DB is the source of truth; the config value is only ever a seed.
  - **Envelope (per-object DEK)**: every new object is encrypted with its own random Data Encryption Key (AES-256-GCM streaming); the DEK is wrapped with the current KEK and stored in the on-disk `.metadata` sidecar (`wrapped-dek`, `wrapped-dek-iv`, `kek-version`). Because the crypto material lives in the sidecar — not only in the metadata DB — objects remain recoverable from the filesystem alone given a KEK backup, and future KEK rotation only re-wraps the small DEK instead of re-encrypting object data.
  - **Multi-format reader (full backward compatibility)**: reads branch per object on its stored format — plaintext objects (from key-less deployments) are served as-is, legacy directly-encrypted objects decrypt with KEK version 1 (the former config key), envelope objects unwrap their DEK. Verified by tests covering all three formats plus restart-persistence.
  - **Writes always encrypt**: the `shouldEncryptObject`/`enable_encryption` conditional and the plaintext write path are gone (single and multipart uploads); `storage.enable_encryption` is deprecated and ignored. Folder markers (zero-byte `key/` entries) are exempt — this also fixes a goroutine leak where the encryption pipe for a folder marker was never read.
  - **Sidecar protection**: `UpdateObjectMetadata` previously replaced the whole `.metadata` sidecar with just the caller's keys, silently destroying the encryption entries of an encrypted object (making it undecryptable). It now merges into the existing sidecar and system keys (`encrypted`, `wrapped-dek`, `original-size`, ...) can never be overridden.
  - The misleading per-bucket "disable encryption" toggle path in bucket creation now consistently records `AES256` on every bucket; the console encryption endpoints report encryption as always enabled.
  (`internal/kek/`, `internal/db/migrations/versions.go`, `internal/object/manager.go`, `internal/server/server.go`, `internal/server/console_api.go`)
- **Background encryption worker** — objects stored before encryption became mandatory (deployments that ran without an encryption key) are now converted to envelope encryption automatically in the background. One shared worker walks every bucket/object and acts on each object's state: plaintext → convert (fresh DEK, wrapped with the current KEK); already encrypted → skip. This is the same component that will re-wrap DEKs on future KEK rotations. Safety first, since it rewrites production data: the plaintext is staged to a temp file, the original is only replaced by an atomic rename, the per-key lock serialises against concurrent writers, and **every converted object is read back, decrypted and MD5-compared** before counting as done (with automatic plaintext restore and concurrent-overwrite detection on mismatch). All stored versions of a key are converted, folder/delete markers and multipart-format ETags are handled. Scheduling is **load-aware** (backs off while CPU >60% or RAM >85%) with gentle pacing, and progress is **checkpointed in Pebble** so a restart resumes where it left off; a full pass re-runs daily to catch strays. New Settings → Security block shows live status (running/paused-by-load/done, converted/skipped/failed counters, per-bucket progress bar) with a "Run now" button; endpoints `GET /api/v1/settings/encryption/worker-status` and `POST /api/v1/settings/encryption/worker-run` (global admin only). Translated in all 9 UI languages. Verified end-to-end with the real binary: a plaintext object planted on disk was picked up by a triggered pass, converted (sidecar gained `wrapped-dek`/`kek-version`, disk bytes became ciphertext) and downloaded back byte-for-byte intact. (`internal/object/encryption_migration.go`, `internal/server/encryption_worker.go`, `web/frontend/src/pages/settings/EncryptionWorkerStatus.tsx`)
- **`maxiofs recover` — offline disaster recovery (metadata DB lost, files intact)** — new CLI subcommand that rebuilds the Pebble metadata store from the on-disk object tree, run with the server stopped. Buckets are rediscovered from their `.maxiofs-bucket` markers (owning tenant included), objects and versions from each file's `.metadata` sidecar (original size/ETag, content-type, encryption fields; versions in `bucket/.versions/key/versionID` are rebuilt in order with the newest as latest, and bucket metrics/versioning status are recomputed). With `--recovery-bundle` the encryption keys are restored into the auth SQLite (so the recovered server's bootstrap loads them instead of generating a fresh KEK) and **every envelope object's wrapped DEK is verified to unwrap** before being counted as recovered. Safety: always writes into a FRESH directory laid out exactly like the live store (activate with a single `mv`), refuses a non-empty output, `--dry-run` reports without writing, and the report classifies objects (verified/unverified encrypted, legacy, plaintext) with per-file failures. Verified end-to-end with the real binary: seeded objects → deleted the entire Pebble store AND the SQLite database → `recover` rebuilt 1 bucket / 2 objects with 2 DEKs verified and 1 KEK restored → the server started with "KEK loaded from database" and served both objects decrypted byte-for-byte. (`internal/recovery/`, `cmd/maxiofs/recover.go`)
- **Encryption key (KEK) rotation** — global admins can rotate the encryption key from Settings → Security (or `POST /api/v1/settings/encryption/rotate-kek`). Rotation creates a new key version and makes it current; **object data is never re-encrypted** — the background worker re-wraps each object's small wrapped-DEK to the new version (verified per object: sidecar update is atomic and the unwrapped DEK is compared byte-for-byte), and legacy directly-encrypted objects (pre-envelope) are now fully converted to a current-KEK envelope during the same pass (decrypted with their original key, staged, re-encrypted and read-back-verified, with the original ciphertext as the restore source). Old key versions are deliberately kept forever: they cost nothing, the recovery bundle includes them, and deleting one would orphan any file still referencing it. Safety tie-ins: rotating **resets the recovery-bundle-downloaded flag** so the console banner reappears prompting a fresh bundle download (the old bundle lacks the new key), and the worker pass starts immediately. In a cluster the new key is created cluster-shared, pushed to all healthy peers at rotation time and reconciled every 60s by the global-config sync (nodes offline during the rotation catch up automatically; raw replication degrades per-node to the legacy path until they do). UI: current version + "Rotate key" button with confirmation in Settings → Security, translated in all 9 UI languages. Verified end-to-end with the real binary: upload (v1) → download bundle → rotate → status flips to v2/not-downloaded → worker re-wrapped the sidecar to v2 with **byte-identical object data** → download returns the content intact. (`internal/kek/store.go`, `internal/object/encryption_migration.go`, `internal/server/encryption_recovery_handlers.go`, `internal/cluster/global_config_sync.go`, `web/frontend/src/pages/settings/EncryptionRecovery.tsx`)
- **Ciphertext HA replication (cluster-shared KEK)** — HA write fanout no longer decrypts on the primary and re-encrypts on each replica. On the first cluster join, the initiating node creates a **cluster-shared KEK** (next free version, marked current) and ships every cluster-shared key inside the existing join package (same channel as the JWT secret); each joining node adopts them — keeping its own local v1 for its pre-join objects — and a version conflict (same version, different key material) aborts the join with a clear error. From then on every node wraps new objects with the same KEK, so the fanout sends the **stored ciphertext + sidecar + Pebble metadata as-is** (`X-HA-Raw` headers); the replica stores a byte-identical copy (with full replica bookkeeping: Pebble entry, metrics, implicit folders) and decrypts only on read. Objects not yet eligible (plaintext, legacy direct-encrypted, or wrapped with a node-local KEK) transparently use the previous decrypt/re-encrypt path and converge to raw-replicable as the rotation worker re-wraps them; a replica that cannot decrypt a KEK version declines with 412 and the primary falls back per node. New `cluster_shared` column on `encryption_keys` (migration 17). Verified end-to-end with a real 2-node cluster: join created/adopted KEK v2 on both nodes, an uploaded object landed on the replica with **byte-identical ciphertext and wrapped DEK**, and the replica decrypted it locally with its own key store. (`internal/kek/`, `internal/cluster/ha_object_manager.go`, `internal/cluster/manager.go`, `internal/server/cluster_handlers.go`, `internal/server/cluster_object_handlers.go`, `internal/object/raw_replication.go`)
- **Encryption recovery bundle (Settings → Security)** — the KEK now lives in the database, so a lost database without a key backup would mean losing every encrypted object. Global admins can download a **recovery bundle**: an export of all KEK versions encrypted under an admin passphrase (PBKDF2-SHA256, 310k iterations + AES-256-GCM; the passphrase is never stored). The console shows an informative banner to global admins **until the bundle has been downloaded** (tracked in `system_settings`), linking to the new section in Settings → Security with passphrase + confirmation inputs. `kek.DecryptBundle()` provides the opening side for the future offline `maxiofs recover` tooling. New endpoints `GET /api/v1/settings/encryption/recovery-status` and `POST /api/v1/settings/encryption/recovery-bundle` (global admin only). Translated in all 9 UI languages. Verified end-to-end: fresh server → status not-downloaded → short passphrase rejected (400) → bundle downloads with correct headers → status flips → bundle decrypts and its KEK matches the database's byte-for-byte. (`internal/kek/bundle.go`, `internal/server/encryption_recovery_handlers.go`, `web/frontend/src/pages/settings/EncryptionRecovery.tsx`, `web/frontend/src/components/layout/RecoveryBundleBanner.tsx`)
- **Per-tenant bandwidth throttling** — global admins can cap a tenant's aggregate transfer bandwidth (upload + download combined) so one client can't saturate the link for others. It **throttles (slows), never rejects** — legitimate bursts are smoothed, not failed. All of a tenant's concurrent transfers share one token-bucket budget (`internal/bandwidth`), enforced on S3 object up/downloads and multipart parts (`pkg/s3compat`). Buckets inherit their tenant's limit (global buckets are exempt); changes apply hot without restart. New tenant field `max_bandwidth_bytes_per_sec` (`0` = unlimited) with SQLite migration, tenant API, and a "Max Bandwidth (MB/s)" field in the tenant form (translated in all 9 UI languages). Verified end-to-end: a tenant capped at 2 MB/s transferred 20 MB in ~10 s (vs ~1 s unlimited), the cap updated live, and two concurrent transfers shared the budget. In a cluster the cap is set once and synced to every node via tenant-sync, then enforced per-node. (`internal/bandwidth/`, `internal/auth/`, `pkg/s3compat/`, `internal/server/console_api.go`, `internal/cluster/tenant_sync.go`, tenant UI)
- **Per-bucket storage quota** — buckets can now have their own size and object-count limits, independent of (and in addition to) tenant quotas. This closes the gap where only tenants had quotas, so a **global** bucket (no tenant) — e.g. a dedicated Veeam/backup target — could not be capped. Highlights:
  - **Model**: optional `Quota { maxSizeBytes, maxObjectCount }` on bucket metadata (`0` = unlimited for that dimension), persisted in Pebble. Current usage reuses the already-cached bucket `TotalSize`/`ObjectCount`, so no new counter is introduced.
  - **Enforcement**: uploads that would exceed the cap are rejected with S3 `QuotaExceeded` (both `PutObject` and `CompleteMultipartUpload`, the path Veeam uses) and `403` on the console upload endpoint. Correct handling of overwrites (net delta, not double-counted), versioned buckets (each version counts), the object-count cap (only on new keys), and HA replica writes (counter updated, enforcement bypassed on replicas). For tenant buckets, **both** the bucket quota and the tenant quota apply — the most restrictive wins.
  - **Veeam/SOSAPI exposure**: the SOSAPI `capacity.xml` a client reads per bucket now advertises capacity with precedence **per-bucket quota → tenant quota → physical disk**, so Veeam (and any SOSAPI client) sees the real bucket limit and its live usage (`Capacity`/`Used`/`Available`), not just the disk or tenant figure.
  - **API + UI**: `GET/PUT/DELETE /api/v1/buckets/{bucket}/quota` (returns quota + live usage; routed to the bucket's owner node in a cluster). New **Quota** tab in bucket settings with a usage bar, translated in all 9 UI languages.
  - **Validation**: a tenant bucket's size quota may not exceed the space assigned to its tenant (`maxSizeBytes ≤ tenant.MaxStorageBytes`), rejected with `400`.
  - **Alerts**: SSE + email notifications (`bucket_quota_warning` / `bucket_quota_critical`, with a matching `bucket_quota_resolved`) fire as a bucket approaches its quota, using the existing warning/critical disk thresholds and escalation-only de-duplication — mirroring the tenant quota alert system.
  - **Cluster**: quota is enforced/aggregated correctly per bucket without cross-node summing (buckets are placed on an owner node and HA replicas are full mirrors, so the owner's local total is authoritative — summing would double-count).

### Changed
- **Removed the vestigial RBAC stub and its SEC-03 startup warning** — the `CheckPermission`/`CheckBucketPermission`/`CheckObjectPermission` chain was unreachable dead code from a superseded design: its only entry point (`AuthorizeRequest`) had no callers, while fine-grained authorization has been enforced since v1.4.0 by the live stack (roles + tenant scoping, the 11-capability role system on both S3 and console, bucket policies with Condition evaluation, and ACLs). The startup warning misled operators into believing permissions were inactive. `GetS3Action`/`GetResourceARN` are kept as primitives for the future IAM policy engine. (`internal/auth/manager.go`, `internal/auth/s3auth.go`, `SECURITY_AUDIT.md`)
- **Removed the broken `make docker-cluster` targets** — `docker-cluster` and `docker-cluster-monitoring` referenced a `cluster` compose profile that never existed, so they silently started a single node while the Makefile and docs promised a 3-node cluster at :8081/:8083/:8085. Beyond being broken, a multi-node cluster inside one Docker host provides no real HA (shared disk/power/kernel) — the targets and their documentation are gone, and CLUSTER.md now carries explicit guidance for containerized deployments (one node per physical host, local volume pinning, port 8082 between hosts). (`Makefile`, `docker/README.md`, `docs/DEPLOYMENT.md`, `docs/CLUSTER.md`)
- **Bucket creation no longer shows an encryption opt-out** — the "enable encryption" checkbox in the create-bucket dialog was misleading: unchecking it never stored anything in plaintext (server-side encryption is always on). The Encryption tab is now informational only, and the creation request always records `AES256` on the bucket. (`web/frontend/src/pages/buckets/create.tsx`)

### Fixed
- **Console: the notification stream now reconnects automatically** — the SSE connection that delivers disk/quota/bucket alerts to the console had no reconnection: a dropped stream (server restart, reverse-proxy idle timeout, network blip) silently ended alert delivery for the rest of the session until a full page reload. The hook now reconnects with exponential backoff (1s → 30s max, reset after a successful connection). Also hardened the four `dangerouslySetInnerHTML` translation renders to route through the shared HTML sanitizer (i18n runs with `escapeValue: false`, so any future user-influenced interpolation in those keys would have reached the DOM unescaped), and throttled the idle-timer's `localStorage` activity writes to once per second (previously a synchronous write per mousemove). (`web/frontend/src/hooks/useNotifications.ts`, `web/frontend/src/hooks/useIdleTimer.ts`, `web/frontend/src/lib/modals.tsx`, `web/frontend/src/components/BucketIntegrityModal.tsx`, `web/frontend/src/components/ObjectLockConfigModal.tsx`)
- **Windows: a transient sharing violation no longer orphans a deleted object** — on Windows, an external scanner (antivirus, search indexer) briefly holding a just-written file without `FILE_SHARE_DELETE` made the physical delete fail with "used by another process"; since the metadata had already been removed, the file stayed behind as an orphan and, worse, HEAD/GET found it again through the sidecar fallback — the object appeared to survive its own deletion. The filesystem backend now retries the file removal briefly (5 attempts with backoff) before giving up. Reproduced consistently by the `TestDeleteObjectErrorCases` flake, now 20/20 green. (`internal/storage/filesystem.go`)
- **Legacy HA transfers now preserve client checksums** — objects replicated through the decrypt/re-encrypt path (plaintext/legacy objects, or a replica missing the cluster KEK) lost their `x-amz-checksum-*` value: the replica's metadata had no `ChecksumAlgorithm`/`ChecksumValue`, so `GetObjectAttributes` answered differently depending on which node served it. All legacy senders (write fanout, sync worker, anti-entropy push) and the internal HA GET now forward the checksum headers; the receiver recomputes and validates them, which also catches a corrupted transfer in flight. (`internal/cluster/ha_object_manager.go`, `internal/cluster/ha_sync_worker.go`, `internal/cluster/anti_entropy.go`, `internal/server/cluster_object_handlers.go`)
- **Concurrent KEK rotations no longer collide** — `Rotate`, `EnsureClusterKey` and `AdoptClusterKeys` computed the next key version without serialising against each other, so two concurrent calls (e.g. rotation racing the periodic cluster sync) could pick the same version and fail on the UNIQUE constraint with a cryptic error. Multi-step KEK mutations are now serialised by a store-level write lock, and `AdoptClusterKeys` applies the whole received key set in a single SQL transaction (a mid-way failure can no longer leave a partial adoption). (`internal/kek/store.go`)
- **KEK sync endpoint returns 400 for malformed payloads (409 only for real conflicts)** — every adoption failure was answered with 409 Conflict, including malformed key material that is a plain bad request. A new `kek.ErrKeyConflict` sentinel marks the genuine same-version-different-material case; the handler maps it to 409 and everything else to 400. (`internal/kek/store.go`, `internal/server/encryption_recovery_handlers.go`)
- **KEK distribution after a rotation survives a dropped admin connection** — the post-rotation push to cluster peers ran under the HTTP request's context, so an admin closing the browser mid-request cancelled the distribution (peers kept wrapping with the old version until the next 60s sync). The push now runs under the server context. Also `GlobalConfigSyncManager.Stop()` is idempotent (a second call panicked on double channel close). (`internal/server/encryption_recovery_handlers.go`, `internal/cluster/global_config_sync.go`)
- **A write acknowledged as successful could be invisible in listings** — when the Pebble metadata save failed after the object file was already stored (disk full, I/O error), `PutObject` only logged a warning and returned success: the S3 client got a 200 for an object that no listing would ever show (backup software verifies uploads by listing — a silent gap). The write now fails with an error so the client retries; the stored data file is deliberately kept (on a non-versioned overwrite it is the only copy, and sidecar-only files are exactly what `maxiofs recover` reindexes). The raw HA replica write applies the same contract, so the primary counts the node as unconfirmed and the sync worker retries instead of leaving an invisible replica. (`internal/object/manager.go`, `internal/object/raw_replication.go`)
- **Encryption worker: a stale checkpoint skipped the entire pass** — if the worker was interrupted mid-pass and its checkpointed bucket was deleted before the next run, the resume logic waited forever for a bucket that no longer existed and silently skipped every remaining bucket, ending the pass as "done" with nothing converted. A checkpoint pointing at a nonexistent bucket now falls back to a full scan. (`internal/server/encryption_worker.go`)
- **`maxiofs recover`: objects with an unverifiable DEK no longer disappear from the rebuilt index** — an envelope object whose wrapped DEK failed to unwrap with the bundle keys (e.g. corrupt sidecar entry) was reported as a failure but EXCLUDED from the rebuilt Pebble store, making bytes that still exist on disk invisible in every listing — unrecoverable in practice. Such objects are now indexed (counted as encrypted-unverified) with the failure kept in the report for the operator. (`internal/recovery/recovery.go`)
- **`maxiofs recover`: KEK restore could leave two keys marked current** — restoring a bundle into a database that already held encryption keys with its own current marker inserted the bundle's current flag alongside the existing one, leaving two `is_current=1` rows and making the server's bootstrap pick one non-deterministically. Bundle keys are now always inserted non-current and the marker is applied once, transactionally: the database's existing current key wins (it is more recent than any external bundle); the bundle's current is applied only when the database had none. (`internal/recovery/recovery.go`)
- **A quota-rejected overwrite destroyed the existing object** — on a non-versioned bucket, `PutObject` stored the new bytes over the final object path BEFORE validating the tenant/bucket quota; when the quota rejected the write, the cleanup deleted that path — which by then held the only copy, so the client got a quota error AND lost the object it was updating (its stale Pebble entry pointed at nothing). The quota check now runs before the store step (the original size is already known from the streaming stage), so a rejection leaves the existing object completely untouched. Regression tests cover both the bucket-quota and tenant-quota paths. (`internal/object/manager.go`)
- **Crash window in object overwrites could leave an undecryptable object** — the filesystem backend committed an overwrite as two consecutive renames (sidecar first, then data). A crash or failure between them left the NEW sidecar (new DEK) over the OLD bytes — undecryptable under envelope encryption — and a failed data rename deleted the previous object's sidecar entirely. Writes now use a staged-sidecar two-phase commit: the new sidecar is staged at `<object>.metadata-staging`, the data is renamed into place (data commit), then the stage is renamed over the final sidecar (metadata commit). A crash at any point is resolved deterministically on the next access: if the stored bytes match the staged etag the metadata commit is completed (roll forward), otherwise the stage is discarded and the old pair remains fully intact (roll back). Per-path striped locks keep concurrent readers from resolving an in-flight stage; staged files are excluded from listings and from `maxiofs recover`; object keys ending in `.metadata`/`.metadata-staging` (which always collided with sidecar files) are now rejected. (`internal/storage/filesystem.go`, `internal/object/manager.go`, `internal/recovery/recovery.go`)
- **Replicas and recovered stores no longer falsify `LastModified` (perpetual anti-entropy churn on multipart objects)** — the Pebble store stamped `LastModified = now` on every metadata write, so an HA replica recorded its receive time instead of the primary's modification time, and a `maxiofs recover` rebuild stamped recovery time instead of the original sidecar timestamp. Consequences: the anti-entropy scrubber's multipart comparison (size + mtime ±1s; multipart ETags can't be checksum-compared) flagged every replicated multipart object as divergent on every cycle and ping-ponged full re-transfers forever, LWW could prefer a stale copy just because it was written locally later, and lifecycle timers reset after recovery. Pebble now preserves a caller-provided `LastModified` (stamping only when unset), raw HA transfers already carry the primary's metadata, legacy HA transfers propagate it via a new `X-HA-Last-Modified` header (set by write fanout, sync worker and anti-entropy push; returned by the HA GET endpoint and honored by both pull paths), and recovery keeps the sidecar's original timestamp. (`internal/metadata/pebble_objects.go`, `internal/object/manager.go`, `internal/cluster/ha_object_manager.go`, `internal/cluster/ha_sync_worker.go`, `internal/cluster/anti_entropy.go`, `internal/cluster/stale_reconciler.go`, `internal/server/cluster_object_handlers.go`)
- **HA object replication requests were 404ing against the dedicated cluster port** — every HA sender (write fanout, HA sync worker, anti-entropy repair, stale-node reconciler) targeted `/api/internal/ha/...`, but since the dedicated inter-node port was introduced those routes have been registered under `/api/internal/cluster/ha/...`. Result: synchronous write fanout, catch-up sync and anti-entropy repair silently failed with 404 on every call (with `factor=2` zero replica confirmations are required, so writes still "succeeded" without ever replicating — matching previously observed ghost sync errors). All 13 sender URLs now use the registered prefix; verified with a real 2-node cluster where fanout previously 404'd and now lands replicas. - **S3 API → console redirect now honors `public_console_url`** (surfaced via GitHub issue #6) — when a browser hits the S3 API, it is redirected to the web console; that redirect hardcoded the internal console port (`8081`) and only used `public_console_url` when `X-Forwarded-*` proxy headers were present. - **Orphaned combined file on multipart upload metadata failure** — if writing final object metadata (`PutObjectVersion` / `PutObject`) failed after `CompleteMultipartUpload` had already physically combined all parts into the destination file, the combined file was left on disk indefinitely with no metadata record, accumulating unrecoverable storage. The failure path now deletes the orphaned file via `storage.Delete` before returning the error. (`internal/object/manager.go`)
- **Lifecycle delete-marker cleanup now has tests that guard against infinite-loop regression** — `processExpiredDeleteMarkers` deletes expired delete markers by calling `DeleteObject` with the marker's specific `versionID`. Without a `versionID`, calling `DeleteObject` on a versioned bucket creates a *new* delete marker instead of removing the existing one, which would cause the lifecycle worker to loop indefinitely. Two tests were added: one that asserts the exact `versionID` is forwarded to `DeleteObject` (preventing the unversioned-delete regression), and one that verifies the worker handles a concurrent deletion (the marker is already gone when the worker processes it) gracefully without panicking. (`internal/lifecycle/worker_test.go`)
- **HA replica writes no longer enforce tenant quota limits** — the primary validates and increments the tenant quota before fanning a write out to replicas. Replicas called the full `PutObject` path including `CheckTenantStorageQuota`, which could reject a replicated write if the replica's local counter was already near the limit (e.g., because the primary incremented it in a prior cycle). Replicas must accept the write unconditionally (the primary already validated it) and only update their own local usage counter so they have an accurate view if they later become primary. A new `WithBypassQuotaEnforcement` context function in the object package is called by `cluster.WithHAReplicaContext`; `PutObject` checks this flag and skips `CheckTenantStorageQuota` while still calling `IncrementTenantStorage`. (`internal/object/manager.go`, `internal/cluster/ha_object_manager.go`)
- **`computeMultipartETag` now rejects part ETags that are not valid 32-character MD5 hex strings** — previously, when `hex.DecodeString` failed on a part ETag it silently fell back to `md5.Sum([]byte(etag))`, producing a different ETag than the S3 specification requires and masking data corruption (e.g., a truncated or misformatted ETag stored during an earlier bug). The fallback is removed; the function now returns an explicit error if any part ETag is not exactly 32 hex characters or cannot be decoded, preserving the integrity of the final object's ETag and making corruption detectable at upload-completion time rather than silently. (`internal/object/manager.go`)
- **Bucket metrics after versioned PUT no longer race against concurrent writes** — `updateBucketMetricsAfterPut` contained a fallback path for the versioned-bucket branch: when `existingObjBeforeSave` was nil it re-queried `GetObjectVersions` after the save to decide whether to increment the object count or only adjust the size. Because the query ran after the save, a concurrent `PutObject` on the same key could insert another version between the save and the query, making the total version count appear ≥ 2 even for a first-time upload — the size would be adjusted instead of the object count being incremented, silently undercounting the bucket. The fallback is removed; all callers are expected to supply `existingObjBeforeSave` captured before the save (the primary production call site at `PutObject` already did this correctly at line 617). The coverage test for additional versions was updated to fetch the pre-save state from the metadata store before the second `PutObject` call. (`internal/object/manager.go`, `internal/object/manager_final_coverage_test.go`)
- **Replication credential decryption failures now surface as an error with an explicit log** — `getRule` and `GetRule` called `decryptCredential` and returned its result without additional logging; if the key had rotated or the ciphertext was corrupt, the caller received a generic wrapped error with no indication of the root cause. A new `decryptAndValidateCredential` wrapper logs at `Error` level before returning a typed `ErrDecryptionFailed` sentinel, and additionally validates that an encrypted (enc1:-prefixed) credential decrypts to at least 8 characters — a partial plaintext write during a previous save could otherwise produce a valid but useless short string. Legacy plaintext credentials (no enc1: prefix) are returned as-is to preserve backwards compatibility. (`internal/replication/credentials.go`, `internal/replication/manager.go`, `internal/replication/worker.go`)
- **Lifecycle worker now paginates object expiration so objects beyond the first page are cleaned up** — `processObjectExpiration` fetched at most 10,000 objects in a single call and stopped, leaving older objects in buckets with more than 10,000 entries permanently unexpired. The worker now loops with a pagination marker until `IsTruncated` is false, processing 1,000 objects per page. A context-cancellation check at the top of each loop iteration ensures the worker stops promptly on server shutdown. (`internal/lifecycle/worker.go`)
- **`DeleteBucket` empty-check and metadata delete are now protected against late object writes** — the previous implementation called `isBucketEmpty` and `DeleteBucket` as two separate store calls with no lock in between, so a concurrent `PutObject` could create object metadata after the empty-check passed but before the bucket key was removed. A first fix moved the check into `DeleteBucketIfEmpty`, but the active Pebble v2 store still performed a prefix scan and delete as separate Pebble operations, leaving a smaller in-process race. Pebble now serializes bucket deletion against object metadata writes (`PutObject`, `PutObjectVersion`, and completed multipart metadata commits) with a per-bucket mutation lock and rejects late writes to buckets deleted during the process lifetime. Added a regression test that verifies a write after `DeleteBucketIfEmpty` fails with `ErrBucketNotFound` and does not leave orphan object metadata. BadgerDB is no longer part of the codebase, so this protection is implemented only in the Pebble-backed metadata store. (`internal/metadata/store.go`, `internal/metadata/pebble_store.go`, `internal/metadata/pebble_objects.go`, `internal/metadata/pebble_multipart.go`, `internal/metadata/pebble_test.go`, `internal/bucket/manager_impl.go`)
- **Replication queue race — same object could be replicated twice** — when the in-memory worker channel was full, `loadPendingItems` released the claim back to `pending` immediately. The next loader tick would re-claim and re-enqueue the same item, so a worker could process it a second time before the first worker finished. Fixed by keeping the claim in `retrying` state instead of releasing it; a new `resetStaleClaimedItems` sweep runs at the start of each loader tick and automatically recovers any item that has been in `retrying` for more than 60 seconds without being picked up by a worker (e.g., because the channel drained faster than the process could proceed). `claimQueueItem` now also stamps `processed_at = CURRENT_TIMESTAMP` so the staleness threshold is always well-defined. (`internal/replication/manager.go`)
- **`CreateBucket` rollback now also reverts the ACL when storage marker creation fails** — when the storage `.maxiofs-bucket` marker `Put` failed, the rollback deleted the bucket metadata entry but left the ACL key (`acl:bucket:<tenantID>:<name>`) in the store, creating an orphaned ACL record that would be silently applied to any future bucket created with the same name. A new `DeleteBucketACL` method on `acl.Manager` deletes the ACL key (ignoring `ErrNotFound` to remain idempotent), and the rollback path now calls it immediately after the metadata rollback so the store is fully clean on failure. (`internal/acl/interface.go`, `internal/acl/manager.go`, `internal/bucket/manager_impl.go`)
- **Lifecycle prefix filter now applied consistently to all action paths** — `processAbortIncompleteMultipartUploads` listed all in-progress multipart uploads for a bucket and aborted any that exceeded `DaysAfterInitiation`, regardless of the rule's prefix filter. This meant lifecycle rules scoped to a prefix (e.g., `logs/`) would abort uploads for any key in the bucket. The function now skips uploads whose key does not start with the configured prefix, consistent with how `processObjectExpiration`, `processNoncurrentVersionExpiration`, and `processExpiredDeleteMarkers` already apply the filter. (`internal/lifecycle/worker.go`)
- **`CompleteMultipartUpload` now defaults Content-Type to `application/octet-stream` when not set** — if the initiating `CreateMultipartUpload` request did not include a `Content-Type` header, the assembled object was stored and returned with an empty `ContentType` field, which caused subsequent `GetObject` responses to omit the `Content-Type` header. The returned object, the encrypted storage metadata, and the unencrypted storage metadata all now fall back to `application/octet-stream` when the multipart metadata does not include a content-type, matching the behavior of regular `PutObject`. (`internal/object/manager.go`)
- **Nil S3 client from replication factory no longer causes a panic** — `replicateObject` and `replicateDelete` called `w.s3ClientFactory(...)` and immediately used the returned `S3Client` interface value without checking for nil. A factory that returns a nil interface (possible in test overrides or future custom factories) would cause a nil-pointer panic on the first method call. Both functions now check the result and return a descriptive error instead of panicking. (`internal/replication/worker.go`)
- **Audit log failures are now surfaced as Warn logs instead of being silently discarded** — several audit call sites still discarded `auditManager.LogEvent` errors with `_ = ...`, so a failed audit write (DB full, I/O error, SQLite contention) could be invisible in compliance-sensitive deployments. The bucket, auth, replication-rule, disk-alert, cluster-node-alert, and integrity audit paths now route through non-blocking helpers that log failures at `Warn` level instead of suppressing them. A follow-up scan confirms there are no remaining discarded `LogEvent` calls in `internal/auth`, `internal/server`, or `internal/bucket`. (`internal/bucket/manager_impl.go`, `internal/auth/audit_helpers.go`, `internal/server/console_api_replication.go`, `internal/server/cluster_node_alerts.go`, `internal/server/disk_alerts.go`, `internal/server/object_integrity_handler.go`, `internal/server/integrity_scrubber.go`)
- **HA metadata fanout now retries with per-attempt timeout and logs divergence on permanent failure** — `fanoutMetadata` dispatched a single goroutine per replica with no timeout or retry. If a transient network error hit during the fire-and-forget, the metadata update was silently lost. Each replica now gets up to 3 attempts with a 5-second per-attempt timeout (using a background context so the replica goroutine outlives the original request) and 200 ms / 400 ms inter-attempt delay. If all retries fail, the failure is logged at `Error` level identifying the replica and op so operators can detect metadata divergence from logs. (`internal/cluster/ha_object_manager.go`)
- **Last-survivor protection now counts only healthy nodes** — `markDeadIfSafe` checked `health_status != dead` (including UNAVAILABLE and DEGRADED nodes) when deciding whether marking a node dead would drop the cluster below the replication factor. A cluster with many unavailable nodes could have its count inflated above `factor`, allowing the last truly healthy node to be marked dead. The query now counts `health_status = healthy AND id != candidate` so only write-capable nodes are counted. (`internal/cluster/dead_node_reconciler.go`)
- **Anti-entropy pull now logs when an object is absent on the peer instead of silently returning** — `pullObjectFromPeer` returned `nil` on HTTP 404 with a code comment ("race; skip") but no log output. Operators had no way to see that a divergence had resolved itself remotely. A short `Info` log is now emitted identifying the peer, bucket, and key so the event appears in the scrubber trace. (`internal/cluster/anti_entropy.go`)
- **Anti-entropy ETag tie now resolved deterministically instead of being left unresolved indefinitely** — `lwwClassify` returned `actNone` when two replicas had the same `LastModified` timestamp but different ETags, leaving the divergence in the "found but not fixed" state forever (visible in scrub metrics as a perpetual unfixed divergence count). The function now uses lexicographic ETag order as a stable tie-breaker: the node with the smaller ETag defers to its peer (pulls), the node with the larger ETag wins (pushes). Both sides reach the same conclusion independently, so the cluster converges in one scrub cycle. (`internal/cluster/anti_entropy.go`, `internal/cluster/anti_entropy_test.go`)
- **Duplicate dead-node SSE events suppressed when node was already dead** — `markDeadIfSafe` emitted a `node_dead` SSE event unconditionally after the UPDATE, even when the `AND health_status != dead` guard in the SQL meant 0 rows were affected (node was already dead from a concurrent call). The function now checks `RowsAffected()` and returns early without emitting the event if the UPDATE was a no-op. (`internal/cluster/dead_node_reconciler.go`)
- **Storage-pressure SSE callback now runs in a goroutine with panic recovery** — the `storagePressureFn` callback was called directly inside the health-check goroutine. A blocking callback (e.g., a slow SSE write) would stall the health ticker for all nodes in the cluster; a panicking callback would crash the goroutine entirely. The callback is now dispatched in a dedicated goroutine with a `defer recover()` that logs the panic at Error level without propagating it. (`internal/cluster/health.go`)
- **HASyncWorker now holds the sync-running lock through the DB existence check** — `runTrigger` released `mu` before querying the DB for an existing sync job, creating a window where two concurrent `Trigger` calls could both see no running job and start duplicate syncs for the same node. The lock is now held through the DB query; if no job is found, the slot is reserved in `w.running` before the lock is released. If the subsequent DB insert fails, the slot is released. (`internal/cluster/ha_sync_worker.go`)
- **Inter-node HMAC replay window reduced from ±5 minutes to ±30 seconds** — the cluster proxy and authentication middleware accepted signed requests up to 5 minutes old, giving an attacker on the intra-cluster network a large capture-and-replay window. Inter-node clocks are NTP-synchronised; the window is tightened to ±30 seconds (matching common practice for signed API tokens on private networks). (`internal/cluster/proxy.go`, `internal/middleware/cluster_auth.go`)
- **Tenant quota not enforced on versioned-bucket uploads** — `PutObject` skipped `CheckTenantStorageQuota` entirely when versioning was enabled (condition included `!versioningEnabled`), so tenants could upload unlimited data to versioned buckets. For non-versioned buckets the delta between old and new object sizes was correctly checked; for versioned buckets the new version size must be added in full (all versions are preserved). The `!versioningEnabled` guard is removed; the function now branches on versioning to compute the correct size increment before the quota check. (`internal/object/manager.go`)
- **`PutObjectTags` concurrent update race — last write silently discarded** — `PutObjectTags` read the object record, merged the new tags, and wrote the result back without holding the per-bucket mutation lock. Two concurrent tag updates on the same key could both read the pre-merge state and one set of tags would be lost. All other write paths (`PutObject`, `PutObjectVersion`, multipart completion) already hold `getBucketMutationMutex` for their read-modify-write. The mutex is now acquired at the top of `PutObjectTags` as well. (`internal/metadata/pebble_objects.go`)
- **OAuth tokens exposed in redirect URL — replaced with server-side one-time code exchange** — `handleOAuthCallback` placed the access token and refresh token directly in the redirect URL query string (`?token=…&refresh_token=…`). Tokens appeared in server access logs, browser history, proxy logs, and any `Referer` header on the next navigation, violating RFC 6819 §4.2.2. The callback now generates a cryptographically random 32-byte hex one-time code (TTL 60 s, stored in a `sync.Map` on the server) and redirects with only `?code=<code>`. A new public endpoint `GET /api/v1/auth/oauth/exchange-code` atomically consumes the code (`LoadAndDelete`) and returns the token pair in the JSON response body, where it is not logged or cached by intermediaries. The frontend `oauth-complete.tsx` is updated to call the exchange endpoint instead of reading tokens from the URL. (`internal/server/console_idp.go`, `internal/server/server.go`, `internal/server/console_api.go`, `web/frontend/src/pages/auth/oauth-complete.tsx`)
- **Deactivated users bypass JWT validation** — `ValidateJWT` looked up the user by username and returned it without checking `user.Status`, so an admin deactivating a user had no immediate effect: the user's access tokens (up to 15-min TTL) and refresh tokens (up to 24-h TTL) remained valid. `ValidateCredentials` and `ValidateConsoleCredentials` already checked `user.Status != UserStatusActive`. The same check is now applied in `ValidateJWT`, returning `ErrUserInactive` immediately and preventing any further use of tokens issued before deactivation. (`internal/auth/manager.go`)
- **Data race in replication rule scheduler — `lastSync` map written from multiple goroutines** — the main `ruleScheduler` goroutine read and deleted entries from `lastSync` while spawned per-rule goroutines wrote to it with no synchronization, constituting a classic Go map race that would trigger a runtime panic under concurrent replication activity. Fixed by recording `lastSync[ruleID] = now` in the main goroutine before spawning the rule goroutine; the goroutine no longer writes to the map, eliminating the shared mutable state entirely without adding a mutex. (`internal/replication/manager.go`)
- **Replication queue cleanup silently discarded database errors** — `cleanupOldQueueItems` discarded the error from `db.ExecContext` with `_`. If the DELETE failed (disk full, locked table), the replication queue would grow without bound with no operator-visible indication. The error is now checked and logged at `Warn` level. (`internal/replication/manager.go`)
- **JSON marshal errors silently ignored in HA metadata fanout** — six `json.Marshal` calls in `HAObjectManager` (for object metadata, tags, ACL, retention, legal-hold, and restore-status operations) discarded their error with `_`. A marshal failure would pass a nil byte slice to `fanoutMetadata`, which would send empty or malformed data to replica nodes, causing silent metadata divergence that would only surface during a subsequent anti-entropy scrub. Each call now checks the error; on failure it logs at `Warn` level with bucket and key fields and skips the replica fanout rather than sending bad data. (`internal/cluster/ha_object_manager.go`)
- **`Stop()` panics on double-call in lifecycle worker and deletion-log sync manager** — both `Worker.Stop()` and `DeletionLogSyncManager.Stop()` closed the `stopChan` unconditionally, so a second call (possible if the HTTP server and the context cancel race during shutdown) would panic with "close of closed channel". Added `stopOnce sync.Once` to both structs; both `Stop()` methods now wrap the `close()` inside `stopOnce.Do(...)`. (`internal/lifecycle/worker.go`, `internal/cluster/deletion_log.go`)

## [1.4.0] - 2026-05-03

### Added
- **Role capabilities system — service-level permission matrix stored entirely in the database** — previously all users shared the same implicit permission set: any authenticated user with the `user` role could create buckets, delete buckets, upload and delete objects, and manage policies. There was no mechanism to restrict what a specific user or role could do at the service level, making it impossible to create read-only or limited-access users without relying exclusively on per-bucket ACLs. The new capabilities system introduces a two-layer model on top of the existing `bucket_permissions`: *role defaults* (what a role allows by default) and *per-user overrides* (explicit allow/deny set by an admin on top of the role default). Eleven capabilities are defined: `bucket:create`, `bucket:delete`, `bucket:configure`, `bucket:manage_policy`, `object:upload`, `object:download`, `object:delete`, `object:manage_tags`, `object:manage_versions`, `console:access`, `keys:manage_own`. Resolution order for every capability check: (1) explicit user override deny → false; (2) explicit user override grant → true; (3) role `admin` safety net → true; (4) role default in `role_capabilities` table → true/false; (5) default → false. The `admin` role always has all capabilities regardless of the table, so global admins can never accidentally lock themselves out. (`internal/auth/capabilities.go`, `internal/db/migrations/migration13_capabilities.go`)
- **`role_capabilities` and `user_capability_overrides` database tables** — migration 13 creates two new SQLite tables. `role_capabilities (role TEXT, capability TEXT, PRIMARY KEY(role, capability))` stores the per-role defaults and is seeded at migration time with opinionated defaults: `admin` gets all capabilities; `user` gets all except `bucket:manage_policy`; `read` gets `object:download`, `console:access`, `keys:manage_own`; `readonly` gets `object:download`, `console:access`; `guest` gets `object:download` only. `user_capability_overrides (id TEXT, user_id TEXT, capability TEXT, granted INTEGER, granted_by TEXT, created_at INTEGER, UNIQUE(user_id, capability))` stores per-user overrides created by admins. All defaults are database-stored and editable by a global admin at runtime — no hardcoded values in Go code. ### Fixed
- **Release and package builds now strip debug symbols consistently** — `build`, `build-server`, `build-all`, platform-specific builds, Docker images, Debian source builds, `.deb`, and RPM package paths now use stripped Go linker flags (`-s -w`), `-trimpath`, and `CGO_ENABLED=0` for cleaner, smaller binaries without embedded local source paths or accidental native-library dependencies. The `.deb` targets also no longer reference the undefined `FLAGS_RELEASE` variable, which previously caused Debian packages to be built without release linker flags. (`Makefile`, `Dockerfile`, `debian/rules`)
- **Multipart uploads stored all HTTP request headers as object user metadata** — `CreateMultipartUpload` iterated over every header in the incoming request and stored them all in the upload's metadata map without filtering. When `CompleteMultipartUpload` committed the final object, that unfiltered map was used as both the storage metadata and the user metadata (`x-amz-meta-*`) of the resulting object. Any client using the AWS SDK or CLI would therefore permanently embed `Authorization`, `User-Agent`, `Amz-Sdk-Invocation-Id`, `X-Amz-Date`, `X-Amz-Content-Sha256`, and other internal request headers into the object record, exposing sensitive values (including the full SigV4 authorization token) via `HeadObject` to any caller with read access. Fixed by replacing the raw header loop in `CreateMultipartUpload` with `extractMetadataFromHeaders` — the same helper used by `PutObject` — which whitelists only `Content-Type`, `Content-Disposition`, `Content-Encoding`, `Cache-Control`, `Content-Language`, and `x-amz-meta-*` headers. `x-amz-acl` is additionally preserved as an internal-only key in the upload metadata (consumed by `CompleteMultipartUpload` to apply the canned ACL after the object is written) and excluded from user metadata by `filterStorageMetadataKeys` so it does not appear in `HeadObject` responses. The bug was present since the multipart upload feature was introduced and is not specific to any single release. (`internal/object/manager.go`)
- **Frontend inactivity timeout no longer treats background token refresh as user activity** — token refreshes updated `last_activity` and dispatched a keep-alive event consumed by the idle timer, allowing an inactive browser tab to stay authenticated indefinitely as long as refresh succeeded. The idle timer now advances only on real browser interaction events, while active users still keep their session alive through normal UI activity. (`web/frontend/src/lib/api.ts`, `web/frontend/src/hooks/useIdleTimer.ts`, `web/frontend/src/__tests__/useIdleTimer.test.ts`)
- **Tenant-scoped Object Lock and legal-hold console actions now preserve tenant context** — several frontend API helpers dropped `tenantId` for object legal hold, object ACL updates, and Object Lock configuration updates, so global admins managing tenant buckets could hit the wrong bucket scope or receive false failures. Those calls now propagate `tenantId`, and the Object Lock modal invalidates the tenant-specific bucket query after saving. (`web/frontend/src/lib/api.ts`, `web/frontend/src/pages/buckets/[bucket]/index.tsx`, `web/frontend/src/pages/buckets/[bucket]/settings.tsx`, `web/frontend/src/components/ObjectLockConfigModal.tsx`)
- **Bucket replication settings now resolve the correct source bucket context without constraining destinations** — the console replication endpoints and frontend helpers did not carry `tenantId` when a global admin opened a tenant-scoped bucket, so listing or managing rules could target the global bucket namespace instead of the tenant bucket. The tenant context now applies only to the MaxIOFS source bucket lookup; destination endpoint, bucket, region, and credentials remain fully S3-compatible and can point to MaxIOFS, AWS S3, MinIO, or any object server. The frontend replication helpers also unwrap the console API's standard `{success,data}` envelope correctly, and the replication rule form now preserves `enabled: false` instead of forcing saved rules back to enabled. (`internal/server/console_api_replication.go`, `internal/server/server_test.go`, `web/frontend/src/lib/api.ts`, `web/frontend/src/pages/buckets/[bucket]/settings.tsx`)
- **Frontend object operations now encode reserved characters in console API routes** — several object helpers interpolated object keys or tenant IDs directly into URLs, so keys containing reserved URL characters such as `?`, `#`, `%`, or spaces could be truncated or routed incorrectly from the web console. Console object reads, deletes, renames, tags, shares, version operations, and presigned URL generation now encode path/query components consistently. (`web/frontend/src/lib/api.ts`)
- **OAuth completion cookies now work on HTTP console deployments** — the OAuth callback always wrote `auth_token` and `refresh_token` cookies with the `Secure` attribute, unlike the normal login flow. ## [1.3.0] - 2026-04-21

### Added
- **Dedicated cluster inter-node port 8082** — all cluster coordination routes (token validation, CSR signing, node registration, entity sync, JWT secret exchange) now run on a dedicated HTTP server bound to `:8082` (configurable via `cluster_listen`). Previously these routes were multiplexed with the S3 API on port 8080, which caused routing conflicts, prevented independent firewall rules, and meant that any node that had not yet initialized a cluster (no `IsClusterEnabled()` at startup) could not participate in a join flow because the routes did not exist yet. Separating the concerns gives operators a clear boundary: port 8080 is S3 client traffic, port 8081 is the web console, and port 8082 is internal cluster coordination — never exposed to end users. (`internal/server/server.go`, `internal/config/config.go`)
- **New config option `cluster_listen`** — controls the bind address for the cluster inter-node server. Default: `:8082`. Set to `127.0.0.1:8082` to restrict inter-node communication to the loopback interface (single-machine test clusters). (`internal/config/config.go`, `docs/CONFIGURATION.md`)
- **Replication `allow_internal_endpoints` config option** — new `replication.allow_internal_endpoints` YAML setting disables SSRF protection for replication destination endpoints, allowing connections to private/internal IP ranges (10.x, 172.16.x, 192.168.x, 127.x). Required when replicating between MaxIOFS instances on the same Kubernetes cluster, Docker network, or LAN where endpoints resolve to private IPs. When `false` (default), the existing SSRF-blocking dialer rejects connections to RFC 1918 addresses and the cloud metadata endpoint (169.254.169.254). (`internal/config/config.go`, `internal/replication/types.go`, `internal/replication/s3client.go`, `internal/replication/manager.go`, `internal/server/server.go`, `config.example.yaml`)
- **Realtime replication mode now functional** — `mode=realtime` replication rules are now triggered automatically on every S3 `PutObject` and `DeleteObject` operation. Previously the realtime mode was accepted in the UI and stored in the database, but no hook existed in the S3 handlers to enqueue objects — replication only worked via manual sync or the `scheduled` mode timer. Added `QueueRealtimeObject()` to the replication manager (filters matching rules by `mode=realtime` only) and async hooks in the S3 `PutObject` and `DeleteObject` handlers that call it after the response is sent. The `scheduled` and `batch` modes are unaffected — the existing `ruleScheduler` continues to process those independently. (`internal/replication/manager.go`, `pkg/s3compat/handler.go`, `internal/api/handler.go`, `internal/server/server.go`)
- **HA write quorum — writes now block until a quorum of replicas acknowledge** — previously `fanoutPut` and `fanoutDelete` in `HAObjectManager` fired replica requests in background goroutines and returned immediately, meaning the client received HTTP 200 after only the local write completed. If the local node died before the background fanout finished, data was silently lost with no indication to the client. Rewritten as a synchronous quorum protocol: the quorum threshold is `ceil(factor/2)` confirmations (the local write counts as 1, so `ceil(factor/2)-1` replica acknowledgements are required). New `Manager.ClusterCanAcceptWrites(ctx)` pre-checks that enough healthy non-local nodes are available before the local write begins, saving a write-then-rollback cycle when the cluster is already degraded. If quorum is not reached after all replica goroutines complete, `collectAndCheckQuorum` returns the new `cluster.ErrClusterDegraded` error: the S3 handler maps it to `HTTP 503` with `Retry-After: 30` on `PutObject`, `DeleteObject`, and `CompleteMultipartUpload`. On a PUT or CompleteMultipart quorum failure the local write is immediately rolled back via `Manager.DeleteObject` under a `WithHARollbackContext(ctx)` so the rollback itself is not fanned out to replicas. DELETE quorum failures are not rolled back — a delete is a tombstone and the anti-entropy scrubber (see below) will reconcile the missing tombstone on lagging replicas; the client receives 503 and is expected to retry. `factor=2` special-case: needed replica confirmations = 0, preserving best-effort second-copy semantics. Strict two-copy guarantees require `factor=3`. (`internal/cluster/ha_object_manager.go`, `internal/cluster/manager.go`, `pkg/s3compat/handler.go`, `pkg/s3compat/multipart.go`; tests in `internal/cluster/ha_quorum_test.go`)
- **HA read fallback with ordered retry across replicas** — `GetObject` previously selected a single read node via `SelectReadNode` and surfaced the error directly to the client if that node was unavailable or returned 404. In an HA cluster this caused spurious 404s and connection errors even when the requested object existed on other replicas. Replaced with `Manager.SelectReadNodes`, which returns an ordered list of all ready replicas sorted by `latency_ms` ascending → `priority` ascending → `name`, then rotated by an atomic `readCounter % N` to preserve round-robin load distribution while still providing a deterministic per-request retry path. New `Manager.TryProxyRead(ctx, w, r, node)` peeks the replica's HTTP status before committing anything to the response writer: `2xx`/`3xx`/non-404 client errors (401/403/412/416) are streamed to the client and short-circuit the retry loop; `404` closes the response and tries the next candidate (object not yet synced on this replica); `5xx` or transport failure closes the response, tries the next candidate, and flips the failing node to `Unavailable` via `markNodeUnavailable`. The `GetObject` S3 handler iterates the candidate list and falls through to a local read on a full miss. Mid-stream failures after a 200 response has been started surface as truncated bodies — those are unavoidable once bytes are committed. The old `SelectReadNode` is kept as a deprecated thin wrapper for other callers. (`internal/cluster/manager.go`, `pkg/s3compat/handler.go`; tests in `internal/cluster/ha_read_test.go`)
- **Anti-entropy scrubber — continuous background reconciliation across all HA replicas** — added `AntiEntropyScrubber` in `internal/cluster/anti_entropy.go`. One goroutine per node runs a scheduler with a jittered 5–60 minute first-run delay then a configurable cycle interval (`ha.scrub_interval_hours`, default 24 h). Each cycle iterates all buckets in randomized order (so the full keyspace is covered within one interval rather than alphabetically-biased detection) and processes objects in batches of `ha.scrub_batch_size` (default 500). Per batch, the scrubber calls `POST /api/internal/ha/checksum-batch` on every healthy peer and reconciles divergences using Last-Writer-Wins (LWW): a peer missing the object or with an older ETag receives a push via the existing `PUT /api/internal/ha/objects/{key}` endpoint under `WithHAReplicaContext`; a peer with a newer ETag triggers a pull (GET from peer, local PUT under replica context). Multipart objects (ETag contains `<md5>-N`) skip ETag comparison and rely on existence + size + 1 s mtime tolerance instead of expensive whole-file recompute. Identical timestamps with different ETags are logged for manual triage and not auto-fixed. The scrubber is crash-safe: after every batch a JSON checkpoint `(bucket_idx, last_key)` is written to Pebble at `ha:scrub:checkpoint` so a restart resumes the same cycle from where it left off. Each run is recorded in the new `ha_scrub_runs` SQLite table (last 30 runs, pruned on completion). Throttled to `ha.scrub_rate_limit` object comparisons per second (default 50) via `time.Sleep` between objects. Four new live-reloadable global config keys: `ha.scrub_enabled`, `ha.scrub_interval_hours`, `ha.scrub_rate_limit`, `ha.scrub_batch_size`. Status exposed via `GET /cluster/ha/scrub-status` (global admin only): last 10 runs plus in-progress checkpoint snapshot. (`internal/cluster/anti_entropy.go`, `internal/cluster/sync_schema.go`, `internal/server/cluster_object_handlers.go`, `internal/server/cluster_ha_handlers.go`, `internal/server/server.go`, `internal/server/console_api.go`; tests in `internal/cluster/anti_entropy_test.go`)
- **Dead-node redistribution — automatic removal of permanently unavailable nodes from write paths** — introduced `HealthStatusDead` as a new terminal health state alongside `unknown / healthy / unavailable`. A new `unavailable_since` TIMESTAMP column (applied via idempotent `applyDeadNodeMigration`) records the start of a continuous outage; `markNodeUnavailable` and `CheckNodeHealth` use `COALESCE(unavailable_since, ?)` so the timestamp survives repeated failed probes and is cleared on the first healthy transition. Once the outage exceeds `ha.dead_node_threshold_hours` (default 24 h, live-reloadable), the new `DeadNodeReconciler` in `internal/cluster/dead_node_reconciler.go` flips the node to `dead` and calls `HASyncWorker.Trigger()` to start a catch-up sync on remaining nodes — because every healthy node already holds every bucket in N-way replication, no per-bucket replica reassignment is needed. The reconciler loop runs every `ha.redistribution_check_interval_minutes` (default 5 min) with a 30 s jittered first pass and reloads the interval on config change; kill switch via `ha.redistribution_enabled`. Last-survivor protection: if marking a node dead would drop the non-dead node count below the replication factor, the reconciler refuses and writes a human-readable reason into `ha.cluster_degraded_reason`, which is exposed via `GET /cluster/ha/degraded-state` and broadcast over SSE as `cluster_degraded` / `cluster_degraded_resolved` events. Admin drain shortcut: `POST /cluster/nodes/{id}/drain` accepts an optional `{"reason"}` body and immediately flips a node to dead, bypassing the threshold timer. The drain endpoint rejects requests targeting the local node (the responding server cannot safely flip itself to dead mid-call) and nodes already in the dead state. The SSE bridge lives in `internal/server/dead_node_events.go`, which decouples the cluster package from the server via an `EventEmitter` callback registered at startup. (`internal/cluster/types.go`, `internal/cluster/schema.go`, `internal/cluster/sync_schema.go`, `internal/cluster/manager.go`, `internal/cluster/health.go`, `internal/cluster/dead_node_reconciler.go`, `internal/server/dead_node_events.go`, `internal/server/cluster_ha_handlers.go`, `internal/server/server.go`, `internal/server/console_api.go`; tests in `internal/cluster/dead_node_reconciler_test.go`)
- **Storage-pressure health state — graceful degradation under high disk usage** — added `HealthStatusStoragePressure` as a node-level health state between `healthy` and `degraded`. The existing health checker already reads `capacity_total` and `capacity_used` from disk on each probe; it now computes `usage% = used/total*100` and transitions a node to `storage_pressure` when usage exceeds `ha.storage_pressure_threshold_percent` (default 90 %). The node returns to `healthy` only when usage drops below `ha.storage_pressure_release_percent` (default 85 %), providing hysteresis that prevents rapid oscillation. A misconfiguration where `release ≥ threshold` is auto-clamped to `threshold - 5` so the hysteresis band is never zero. Nodes in `storage_pressure` are excluded from write target selection (`GetHealthyNodes` uses a strict `= healthy` filter) but remain eligible for reads (`GetReadyReplicaNodes` accepts `IN (healthy, storage_pressure)`) because they still hold valid data. `HealthStatusStoragePressure` never overrides `dead`, `unavailable`, or `degraded` — the transition only fires for reachable low-latency nodes. Two new live-reloadable global config keys: `ha.storage_pressure_threshold_percent` and `ha.storage_pressure_release_percent`. SSE events `node_storage_pressure` (carries `usage_percent` + `threshold_percent`) and `node_storage_pressure_resolved` are emitted as one-shots on state transitions via `cluster.StoragePressureEmitter` callback and the new SSE bridge in `internal/server/storage_pressure_events.go`. (`internal/cluster/types.go`, `internal/cluster/sync_schema.go`, `internal/cluster/manager.go`, `internal/cluster/health.go`, `internal/server/storage_pressure_events.go`, `internal/server/server.go`; tests in `internal/cluster/storage_pressure_test.go`)
- **Stale-node reconciler — safe catch-up when a partitioned or offline node rejoins** — added `StaleReconciler` in `internal/cluster/stale_reconciler.go`. Runs once at node startup when `is_stale = true` is detected in `cluster_nodes`. Detects one of two modes based on `last_local_write_at`: `ModeOffline` (node was unreachable but did not accept writes — only tombstones need immediate propagation) or `ModePartition` (node was isolated but continued serving writes — a full LWW push of locally-newer entities is performed before the stale flag is cleared). In `ModePartition` the reconciler builds a local `StateSnapshot` (entity IDs + `updated_at` timestamps for tenants, users, access keys, bucket permissions, IDP providers, group mappings) and fetches the corresponding snapshot from each peer via `GET /api/internal/cluster/state-snapshot`, then pushes any entity where the local timestamp is strictly greater. Both modes perform bidirectional tombstone sync: local tombstones absent or newer on the peer are pushed via the existing `deletion-log-sync` endpoint; remote tombstones absent or newer locally are applied with LWW (`EntityIsNewerThanTombstone` prevents overwriting a local entity that was re-created after the deletion). After all entity reconciliation completes, the stale flag is cleared. When the node was previously a ready HA replica (a `ha_sync_jobs` record with `status=done` exists), a further HA object delta sync runs: the reconciler contacts any healthy peer via `GET /api/internal/ha/objects/changed-since` (paginated by bucket), then pulls each changed key with `GET /api/internal/ha/objects/{key}` and writes it locally under `WithHAReplicaContext`, creating a tracked `ha_sync_jobs` entry for progress visibility. A `NewDynamicProxyClient` is used so TLS config changes after construction (e.g. a cluster join after the reconciler was created) are picked up automatically on each request. (`internal/cluster/stale_reconciler.go`, `internal/cluster/proxy.go`, `internal/cluster/manager.go`, `internal/server/server.go`)
- **Inter-node S3 proxy with HMAC authentication** — added a dedicated S3 proxy layer in `internal/cluster/proxy.go` that handles forwarding S3 client requests to the node that owns the requested bucket or object. `ProxyClient` was extended with `NewDynamicProxyClient(getTLS func() *tls.Config)`: unlike the original `NewProxyClient` (which captured TLS config once at construction), the dynamic variant re-evaluates the getter on each request and transparently rebuilds the underlying `http.Client` when the TLS config pointer changes — critical for sync managers and the stale reconciler that outlive cluster join. Two new functions handle inter-node S3 forwarding: `ProxyToNodeAPIURL` derives the target URL from `node.Endpoint` (always an internal IP, avoiding public-cert issues) via `buildInternalS3URL`, copies headers from the original request, and replaces the client's `Authorization` header with cluster HMAC auth; `AddClusterProxyHeaders` computes `HMAC-SHA256(clusterToken, "proxy:{nodeID}:{timestamp}")` and adds `X-MaxIOFS-Node-Hmac` + forwarded user context (`X-MaxIOFS-Forwarded-User`, `X-MaxIOFS-Forwarded-Tenant`, `X-MaxIOFS-Forwarded-Roles`). The receiving node validates the proxy auth via `ValidateClusterProxyAuth`, which checks the `X-MaxIOFS-Proxied: true` header, recomputes the HMAC, and enforces a ±5-minute timestamp window to prevent replay attacks. Loop prevention is enforced at `ProxyClient.ProxyRequest`: if the incoming request already carries `X-MaxIOFS-Proxied: true`, the proxy immediately returns an error instead of forwarding, preventing infinite A→B→A proxy chains. `buildInternalS3URL` extracts the IP from `node.Endpoint` (cluster port) and derives the S3 port from `node.APIURL`'s explicit port if set (skipping defaults 80/443), falling back to 8080 — ensuring inter-node S3 traffic always travels over internal IPs regardless of the configured public API URL. (`internal/cluster/proxy.go`, `internal/cluster/stale_reconciler.go`, `internal/cluster/router.go`, all six `*_sync.go` managers, `internal/server/console_api.go`; tests in `internal/cluster/proxy_test.go`)

### Fixed
- **Cluster sync tables never created — all inter-node synchronization permanently disabled** — `server.New()` called `cluster.InitSchema(db)` to create the cluster membership tables but never called `cluster.InitReplicationSchema(db)`, which is responsible for creating the sync-tracking tables (`cluster_entity_sync`, `cluster_sync_state`, etc.) and the `cluster_global_config` table with default auto-sync flags (`auto_users_sync_enabled=true`, `auto_access_keys_sync_enabled=true`, `auto_tenants_sync_enabled=true`). Every sync manager's main loop queries `cluster_global_config` at startup to determine whether synchronization is enabled; because the table did not exist, the query returned a SQL error that was silently treated as "disabled" and the manager returned immediately on every tick. As a result, access keys, users, tenants, bucket permissions, IDP providers, and group mappings were **never** propagated between cluster nodes in any production deployment. (`internal/server/server.go`, `internal/cluster/sync_schema.go`)
- **Remote nodes always showed 0 GB disk capacity in the cluster node list** — `handleAPIHealth` on the cluster inter-node port wrapped its response in the `APIResponse` envelope (`{"success":true,"data":{"status":"healthy"}}`). The health checker that parses remote node responses reads `capacity_total` and `capacity_used` as top-level flat JSON fields; because both fields were nested inside the `data` object they were never resolved and always defaulted to `0`, making every remote node appear to have no available disk space in the console. Fixed by rewriting `handleAPIHealth` to emit a flat JSON object containing `status`, `capacity_total`, and `capacity_used` directly from `s.systemMetrics.GetDiskUsage()`. (`internal/server/console_api.go`)
- **Inter-node sync requests rejected with TLS certificate errors** — each sync manager (`AccessKeySync`, `UserSync`, `TenantSync`, `BucketPermissionSync`, `IDPProviderSync`, `GroupMappingSync`) created its `proxyClient` (an `http.Client` with the cluster TLS configuration) inside `server.New()`, before cluster initialization had run at runtime. At that point the cluster CA certificate had not yet been loaded, so the already-built clients had no trusted CA and all subsequent HTTPS inter-node sync requests were rejected with TLS verification errors. Fixed by refreshing `m.proxyClient = NewProxyClient(m.clusterManager.GetTLSConfig())` at the start of each manager's `Start()` method, after cluster TLS has been fully initialized. (`internal/cluster/access_key_sync.go`, `internal/cluster/user_sync.go`, `internal/cluster/tenant_sync.go`, `internal/cluster/bucket_permission_sync.go`, `internal/cluster/idp_provider_sync.go`, `internal/cluster/group_mapping_sync.go`)
- **Access keys, users, and tenants unavailable on newly-joined nodes for up to 30 seconds** — after `handleAddClusterNode` registered a new node in the database its `health_status` was `unknown`. `GetHealthyNodes()`, called by all sync manager loops to obtain the list of peers, filters exclusively for `health_status='healthy'` rows. The new node was therefore excluded from every sync iteration until the health-check goroutine ran (30-second interval), validated the node, and updated its status. During this window any S3 client that connected to the new node received `InvalidAccessKeyId` for every request. Fixed by adding `kickstartNewNodeSync()`: immediately after the join sequence completes a goroutine calls `CheckNodeHealth` to mark the node healthy, then calls `tenantSyncMgr.SyncToNode`, `userSyncMgr.SyncToNode`, and `accessKeySyncMgr.SyncToNode` in order to push all existing data to the new node without waiting for the periodic loops. (`internal/server/cluster_handlers.go`, `internal/cluster/user_sync.go`, `internal/cluster/access_key_sync.go`, `internal/cluster/tenant_sync.go`)

- **Cluster join permanently broken: primary node sent its console URL instead of S3 API URL to joining nodes** — `handleAddClusterNode` built the `node_endpoint` payload from `s.config.PublicConsoleURL` (port 8081) instead of `s.config.PublicAPIURL` (port 8080). The joining node stored this as the primary's endpoint and used it for all subsequent inter-node calls (`/api/internal/cluster/validate-token`, `/api/internal/cluster/register-node`, `/api/internal/cluster/sign-csr`, `/api/internal/cluster/nodes`, `/api/internal/cluster/jwt-secret`). All of these routes are registered exclusively on the S3 API port; sending them to the console port returned 404 on every call, making cluster join always fail in production. Only worked in tests because both ports were `localhost` and tests happened to hit the right server. Fixed by moving all cluster routes to the dedicated cluster port 8082. (`internal/server/cluster_handlers.go`, `internal/server/server.go`)
- **Inter-node TLS verification always failed in real multi-node deployments** — `GenerateNodeCert` and `GenerateKeyAndCSR` hardcoded only `localhost` and `127.0.0.1` in the certificate Subject Alternative Names. When nodes communicated via real IP addresses or hostnames (e.g. `192.168.1.10`, `node2.internal`), TLS verification rejected the connection because the address did not match any SAN in the peer certificate. This silently broke all HTTPS inter-node traffic outside of single-machine test setups. Fixed by adding an `endpointURL` parameter to both functions; the hostname or IP is extracted from the URL and appended to the SANs alongside the existing `localhost` entries. - **Cluster inter-node communication and direct console access broken after configuring `public_console_url`** — `setupConsoleRoutes` extracted the path component from `public_console_url` and mounted all console routes (API and SPA) under that subpath. Because cluster nodes communicate with each other directly by IP and port (not through the public URL), any deployment that set `public_console_url` to a domain name or load-balancer address caused every inter-node API call to receive 404, effectively breaking cluster formation, node join, and all intra-cluster coordination. Direct operator access to individual nodes via IP:port was equally broken for the same reason. `spaHandler.ServeHTTP` compounded the problem by also returning 404 for requests whose path did not start with the configured base path. `public_console_url` and `public_api_url` define the public-facing URLs used for link generation, OAuth callbacks, and SPA `<base>` tag injection — they must not control which paths the server listens on; that is the sole responsibility of `listen` and `console_listen`. - **Session always expired after exactly 15 minutes regardless of user activity** — the refresh token issued at login was never stored in the browser because `APIClient.login` read `response.data.refreshToken` (camelCase) while the backend sends `refresh_token` (snake_case). As a result the in-memory `refreshToken` field was always `undefined`, which `TokenManager.setTokens` silently ignored. Every subsequent mechanism that depends on the refresh token — the proactive timer that fires 2 minutes before access-token expiry, the sliding-window background refresh that fires on each API response, and the 401 retry interceptor — found no token and did nothing. The access token (default 15 min) expired, the 401 interceptor had no refresh token to exchange, and the user was logged out. - **Inactivity timeout did not account for tab close / page reload** — the idle timer lived entirely in memory (`useRef`), so closing the browser tab or reloading the page reset the inactivity counter to zero. A user who left the tab open for 10 minutes, closed it, and reopened it an hour later would not be logged out because no timer had been running during the closure. Fixed by persisting `last_activity` to `localStorage` on every user-interaction event (mousedown, keydown, scroll, click, etc.). - **Duplicate folders in S3 clients when listing large buckets** — S3 clients (S3 Browser, AWS CLI, SDKs) that paginate a bucket with `delimiter=/` received the same common prefix (folder name) on multiple pages. Root cause had two parts: (1) `ListObjects` in `manager.go` set `nextMarker` to the common prefix string (e.g. `"folder_1000/"`) when truncating a page that ended on a folder boundary. On the next page the Pebble iterator seeked to that key and, if it existed as a folder marker or if objects inside the folder were encountered, extracted the same common prefix again. (2) When the store's own pagination `nextMarker` fell inside a returned common prefix (large folder with >100 000 objects across multiple 100 k-scan chunks), the same folder was emitted again on the next chunk's page. The function is applied in two places: when truncating by prefix count, and when the store's `nextMarker` falls inside any returned common prefix. (`internal/object/manager.go`)
- **Pebble `ListObjects` / `searchObjectsByScan` skipped all results when marker key was not in store** — the skip-to-marker loop used `objKeyStr == marker` to detect when to start collecting. `SeekGE` positions the iterator at the first key ≥ marker; if the marker is a synthesised sentinel (e.g. `"folder_1001"` produced by `advancePastPrefix`) that does not exist as a real object key, the first key from `SeekGE` is already past the marker, but `started` never became `true` and every subsequent key was discarded. Fixed by promoting `started = true` immediately when `objKeyStr > marker` (SeekGE guarantees the key is ≥ marker, so `>` means the marker was not found and the current key is the correct start), falling through to process the key instead of continuing. The `== marker` branch still skips the marker itself for exclusive-lower-bound semantics. Same fix applied to `searchObjectsByScan`. (`internal/metadata/pebble_objects.go`)
- **`handleCreateBucket` panicked with nil pointer dereference in server tests** — the handler called `s.clusterManager.GetLocalNodeID(r.Context())` unconditionally to assign the HA primary node. In the test environment `s.clusterManager` is nil (cluster is not initialised for unit tests), causing a panic. Every other call site in `console_api.go` guards with `if s.clusterManager != nil`. Added the same guard around the primary-node assignment. (`internal/server/console_api.go`)
- **Bucket migration available when HA replication factor > 1** — the `POST /cluster/buckets/{bucket}/migrate` handler did not check the current replication factor. In HA mode every node already holds a complete copy of all data, so migrating a bucket to a specific node is meaningless and, if `delete_source` was set, would destroy data that is still serving live read traffic. - **Bucket migration `delete_source` option was a no-op** — `executeMigration` in `migration.go` contained a `// TODO: Implement source deletion` comment and no code. Setting `delete_source: true` silently succeeded without removing anything from the source node. ### Fixed
- **Background token refresh triggered immediate logout on transient failure** — the `handleResponse` activity-based background refresh (fired after every successful API call, debounced to once per minute) called `doLogout()` directly when the refresh endpoint returned 401 or 403. This meant that if the refresh token expired between two user actions, the user was silently logged out the next time any API call completed successfully — even though their current access token was still valid for up to another 14 minutes. The race was particularly common when the `security.session_timeout` was set shorter than intended: both the access token and refresh token would share the same short TTL, expire together, and the next background refresh would immediately terminate the session. Fixed by removing `doLogout()` from the background refresh catch path; the 401 error interceptor (which fires only when an actual API request is rejected) remains the sole authoritative logout trigger. Additionally, `setTokens` no longer clears the in-memory refresh token to `null` when called without a second argument (preventing a silent accidental clear if a refresh response ever omits the `refresh_token` field), and `getRefreshToken()` now falls back to localStorage if the in-memory copy is missing. (`web/frontend/src/lib/api.ts`)
- **COMPLIANCE Object Lock retention did not prevent bucket deletion by global admins** — the `handleDeleteBucket` console API handler blocked non-admin users from deleting Object Lock–enabled buckets but allowed global administrators to delete them regardless — including via the `?force=true` path that calls `ForceDeleteBucket`, which bypasses the empty-bucket check entirely. Any user with global admin credentials could destroy WORM-protected data, defeating the guarantee offered to Veeam B&R users and similar backup workloads that rely on COMPLIANCE retention. Fixed by adding a pre-deletion scan (`HasActiveComplianceRetention`) that iterates all object metadata and versioned object entries for the bucket and returns `true` if any record has `Retention.Mode == "COMPLIANCE"` with a future `RetainUntilDate`, or has `LegalHold == true`. If active immutable data is found, the request is rejected with `HTTP 409 Conflict` for **all** callers — including global admins and force-delete. `HasActiveComplianceRetention` was added to the `metadata.Store` interface, its Pebble implementation, the `object.Manager` interface, and `objectManager`. The bucket list UI now disables the delete button for **all** users on Object Lock buckets (previously only blocked non-global-admins) with an updated tooltip explaining why. The `onError` handler in the bucket list no longer offers the force-delete dialog when the server responds with a COMPLIANCE error. (`internal/metadata/store.go`, `internal/metadata/pebble_objects.go`, `internal/object/manager.go`, `internal/server/console_api.go`, `web/frontend/src/pages/buckets/index.tsx`)
- **GOVERNANCE Object Lock bypass ignored when deleting a specific version** — `DeleteObject` reads the `x-amz-bypass-governance-retention: true` header and passes `bypassGovernance=true` to `objectManager.DeleteObject`, but when a `versionId` is present the call is dispatched to `deleteSpecificVersion`, which did not accept the flag. The GOVERNANCE retention check inside `deleteSpecificVersion` always blocked the deletion regardless of the header, so `--bypass-governance-retention` had no effect on versioned deletes. Fixed by threading `bypassGovernance bool` through `deleteSpecificVersion` and applying the same bypass logic that `deletePermanently` already had. (`internal/object/manager.go`)
- **`GET /{bucket}?uploads` (ListMultipartUploads) was captured by ListObjects** — `ListMultipartUploads` was registered on the bucket router after `ListObjects` (which has no query-parameter constraint). gorilla/mux matches routes in registration order, so every `GET /{bucket}?uploads` request was handled by `ListObjects`, which returned a `ListBucketResult` instead of `ListMultipartUploadsResult`. The AWS CLI parsed the wrong XML and showed an empty response with no `Uploads` array even for in-progress uploads. Fixed by moving the `ListMultipartUploads` route registration inside the per-path loop, immediately before `ListObjects`, so the more-specific query-constrained route is evaluated first. (`internal/api/handler.go`)
- **Panic `pebble: closed` on graceful shutdown** — `BucketAccessLogger.Stop()` closed the `done` channel to signal the background goroutine to stop and flush, but returned immediately without waiting for the goroutine to finish. The shutdown sequence then closed the Pebble metadata store while the goroutine was still in `flushEntries` → `PutObject` → `pebbleGet`, causing a `panic: pebble: closed`. Fixed by adding a `sync.WaitGroup` (`wg.Add(1)` before starting the goroutine, `defer wg.Done()` at the top of `run`). `Stop()` now calls `wg.Wait()` after closing the channel, blocking until the goroutine has completed its final flush before Pebble is closed. (`internal/server/bucket_logging.go`)
- **Inventory report ETag was the content length encoded as hex, not an MD5 checksum** — when the inventory generator uploaded a report object it set `ETag: fmt.Sprintf("%x", len(content))`, which produces the byte count in hexadecimal (e.g. `"f5a"` for a 3 930-byte CSV). The integrity scanner compares stored ETags against the actual MD5 of the file content, so every inventory report was flagged as corrupt on the next scrub cycle. Fixed by computing `md5.Sum(content)` and storing it as a plain hex string via `fmt.Sprintf("%x", md5.Sum(content))`. ETags are stored without surrounding double-quote characters in the metadata store (the HTTP `"` quoting is an on-wire convention only); a prior attempt that used `fmt.Sprintf("\"%x\"", ...)` stored literal `"` characters in Pebble, causing the integrity check to always report the object as corrupted even after the MD5 fix. (`internal/inventory/generator.go`)
- **Object deletion order left metadata intact after physical file was removed** — `deletePermanently` deleted the physical storage file in step 1 and the metadata in step 2. If step 2 failed (or if the process crashed between the two steps), the object remained visible in metadata but its data was gone; the integrity scanner would then report it as corrupt on the next scan. Swapped the order: metadata is now deleted first, then the physical file. If the physical deletion fails the file becomes an orphan (harmless, cleaned up by the next bucket directory removal or orphan scrub), which is the safe failure direction. (`internal/object/manager.go`)
- **`isBucketEmpty` silently deleted orphaned physical files as a side effect** — when `DeleteBucket` called `isBucketEmpty` to verify the bucket was safe to remove, the helper walked all physical files and deleted any that had no corresponding metadata entry. This caused unexpected `"Found orphaned physical file without metadata - deleting"` warnings in the logs before the bucket-deleted audit event, making it appear the storage layer was making unilateral deletions outside of any explicit user action. Removed the destructive side effect: `isBucketEmpty` now simply skips files with no metadata (treating them as absent for the empty-check). The `DeleteBucket` `RemoveDirectory` call at the end of the deletion sequence already removes the entire directory including any residual orphaned files. (`internal/bucket/manager_impl.go`)
- **Console UI allowed deleting Object Lock–enabled buckets** — any authenticated user (including non-admins) could delete a bucket that had Object Lock enabled through the console API, bypassing WORM protection. Added a server-side guard in `handleDeleteBucket`: if the bucket has `ObjectLockEnabled: true` and the caller is not a global administrator the request is rejected with `403 Forbidden`. The delete button in the bucket list is also disabled in the UI for Object Lock buckets for non-global-admins, with a tooltip explaining why. (`internal/server/console_api.go`, `web/frontend/src/pages/buckets/index.tsx`)
- **Non-admin users could not see buckets they created** — when a global user (no tenant) created a bucket through the console API, `handleCreateBucket` left `OwnerID` and `OwnerType` empty on the new bucket. `FilterBucketsByPermissions` grants visibility when `bucket.OwnerID == user.ID && bucket.OwnerType == "user"`, so the bucket was invisible to the creator immediately after creation. Fixed by assigning `OwnerID = user.ID` and `OwnerType = "user"` for global non-admin users on bucket creation. (`internal/server/console_api.go`)
- **2FA login failed with HTTP 500 after code was accepted** — `GetUserWith2FA` scanned the `tenant_id` column directly into a `string` variable. `tenant_id` is `NULL` in SQLite for users without a tenant assignment; `database/sql` does not allow scanning `NULL` into a plain `string` and returns an error, which the handler mapped to a 500 response. Fixed by scanning into `sql.NullString` and only assigning `user.TenantID` when `Valid` is true. (`internal/auth/sqlite.go`)
- **Vite compiled frontend with absolute `/assets/…` import paths** — `vite.config.ts` had `base: '/ui/'` hardcoded, so dynamic `import()` chunks were compiled with `/ui/assets/chunk-xxx.js` absolute paths. In a deployment where the console is served under a different prefix, the browser fetched those paths from the S3 API port, which returned a MIME-type error and caused a blank page. Changed to `base: './'` so all asset paths are relative to the HTML document. `embed.go` injects a `<base href="…">` tag that resolves relative imports to the correct runtime prefix regardless of the configured console path. (`web/frontend/vite.config.ts`)

- **`RestoreAlreadyInProgress` returned HTTP 500 instead of 409** — the error code was missing from the `writeError` status-code table, so the server logged it as an internal error and the AWS CLI retried the request 4 times before giving up. Added `RestoreAlreadyInProgress` to the 409 Conflict case. (`pkg/s3compat/handler.go`)
- **`NoSuchConfiguration` (inventory) returned HTTP 500 instead of 404** — same root cause: the error code was not mapped in `writeError`. Added `NoSuchConfiguration` to the 404 Not Found case so `GetBucketInventoryConfiguration` on a missing id returns a clean 404 instead of an internal error. (`pkg/s3compat/handler.go`)
- **`RestoreObject` allowed re-restoring an already-restored object** — the in-progress check only tested `RestoreStatus == "ongoing"` but the handler sets status to `"restored"` (not `"ongoing"`) on success. A second restore call within the expiry window silently succeeded with 200 instead of returning 409. Fixed by also checking `RestoreStatus == "restored" && RestoreExpiresAt.After(now)`. (`pkg/s3compat/handler.go`)

- **Metadata store not closed on shutdown — all recent writes silently lost on restart** — `s.metadataStore.Close()` was missing from the server `Shutdown()` sequence. Pebble uses `pebble.NoSync` for all writes (intentional for performance), meaning writes go to the WAL in memory and are only flushed to disk when `db.Close()` is called. Without this call, every metadata change since the last background compaction flush — object deletes, version promotions, renames, tag updates, bucket config changes — was silently discarded on process exit. On the next startup Pebble replayed only what had been physically written to disk, reverting the database to an earlier state. This caused symptoms like: deleting a file and seeing it reappear after restart, restoring a version (removing a delete marker) only for the marker to come back after restart, and renamed objects reverting to their original key. Fixed by adding `s.metadataStore.Close()` at the end of `Shutdown()`, after all workers have stopped (workers may still write metadata during their own shutdown, so order matters). (`internal/server/server.go`)
- **Session expiry while actively working in the UI** — the access token used a fixed 15-minute window from the moment it was issued, with no sliding behaviour. A user who worked continuously for 15 minutes was silently logged out regardless of their activity, because the token timer was never reset by API calls. Implemented a true sliding-window session: after every successful API response a background token refresh is triggered (debounced to at most once per minute via `lastActivityRefreshAt`). Any action that produces an API call — navigation, bucket operations, uploads, settings changes — now resets the 15-minute window from that moment. The existing proactive refresh timer (fires 2 minutes before expiry) is kept as a safety net for idle tabs where no API calls are in flight. (`web/frontend/src/lib/api.ts`)
- **Proactive refresh silently abandoned on transient failure** — if the proactive background refresh failed due to a network glitch or server hiccup, the `catch` block discarded the error and no retry was scheduled. The token then expired with no further recovery attempt before the 401 interceptor. The proactive refresh now retries every 30 seconds on transient errors (non-401/403 responses) for as long as the current token is still alive. Auth rejections (401/403) are not retried — those mean the refresh token itself is invalid and the next API call's 401 interceptor handles the final logout. (`web/frontend/src/lib/api.ts`)
- **Idle timer logged out user during active file upload** — the idle timer tracks UI events (mouse, keyboard) to detect inactivity. During a large file upload the user is not interacting with the UI, so after 30 minutes of upload progress the idle timer fired `handleIdle` and redirected to login, aborting the transfer. The idle timer now checks an `activeUploadCount` counter before firing: if any upload is in progress the logout is postponed by a full 30-minute interval instead of executing. The counter is incremented when `uploadObject` starts and decremented in `finally` so it is always correct even on error. (`web/frontend/src/lib/api.ts`, `web/frontend/src/hooks/useIdleTimer.ts`, `web/frontend/src/hooks/useAuth.ts`)
- **Console UI allowed deleting Object Lock–protected objects** — the bucket browser did not check an object's retention or Legal Hold status before calling the delete API. Objects marked as immutable by VEEAM B&R appeared locked in the UI (lock icon + retention countdown) but the delete button remained enabled. (`web/frontend/src/pages/buckets/[bucket]/index.tsx`, `web/frontend/src/locales/en/buckets.json`, `web/frontend/src/locales/es/buckets.json`)

### Changed
- **Pebble metadata engine upgraded from v1.1.5 to v2.1.4** — the embedded LSM-tree engine that stores all object and bucket metadata has been updated to the latest stable release. Pebble v1 and v2 use **incompatible on-disk formats**, so the server performs an automatic one-time migration on first start with this version:
  1. The existing Pebble v1 database (`metadata/`) is opened **read-only**.
  2. All key-value pairs are streamed into a temporary v2 directory (`metadata_pebblev2/`).
  3. The directories are swapped atomically: `metadata/` is renamed to `metadata_pebblev1_backup_{timestamp}/` (preserved as a safety backup), and `metadata_pebblev2/` is promoted to `metadata/`.
  4. A sentinel file `metadata/PEBBLE_FORMAT_V2` is written. All subsequent restarts detect it and skip the migration check immediately.

  The migration is fully transparent — no manual steps, no downtime window, no data loss. The backup directory can be deleted after verifying the server operates normally. If the server is killed mid-migration, the next start automatically detects the incomplete state and either retries or completes the rename. Upgrading from BadgerDB (pre-v1.0.0-beta) is also covered: the BadgerDB migration path now writes to Pebble v2 directly and stamps the same sentinel, so the v1→v2 check is skipped for those deployments too.

  **Upgrade path summary**:
  - Pre-v1.0.0-beta (BadgerDB) → first upgrade to **v1.0.0-beta**, which migrates to Pebble v1, then upgrade to this version, which migrates to Pebble v2. Alternatively, upgrading directly to this version also works if the BadgerDB→Pebble v2 direct path runs first.
  - v1.0.0-beta (Pebble v1) → upgrade directly to this version; automatic migration runs on first start.
  - Fresh install → Pebble v2 is created directly, no migration needed.

  (`internal/metadata/pebble_v1_migration.go`, `internal/metadata/pebble_store.go`, `internal/metadata/migration.go`, `internal/server/server.go`, `go.mod`)

- **New config option: `storage.metadata_cache_size_mb`** — controls the size of the Pebble block cache, which is the primary lever for metadata read performance. When MaxIOFS lists a folder or fetches object info, Pebble first checks this in-RAM cache. A cache hit is essentially instant (~0.1 ms); a miss requires a disk read. The cache is populated on first access and persists for the lifetime of the process.

  Default is **256 MB**. Recommended values by workload:

  | Deployment | Objects / bucket | Recommended |
  |------------|-----------------|-------------|
  | Small / development | < 5 000 | 256 MB (default) |
  | Medium | 5 000–20 000 | 512 MB |
  | Veeam B&R / large buckets | 20 000–100 000 | 1 024 MB |
  | Very large deployments | 100 000+ | 2 048 MB+ |

  Configure in `config.yaml`:
  ```yaml
  storage:
    metadata_cache_size_mb: 1024
  ```
  Or via environment variable (no restart of config file needed):
  ```bash
  MAXIOFS_STORAGE_METADATA_CACHE_SIZE_MB=1024
  ```
  **Note**: this is a startup-time setting. It cannot be changed at runtime via the console or API — the cache is allocated once when the database opens. Restart is required after changing it.

  (`internal/config/config.go`, `internal/metadata/pebble_store.go`, `internal/server/server.go`, `config.example.yaml`, `docs/CONFIGURATION.md`)

- **Pebble pre-tuned for write-heavy workloads** — several internal engine parameters were tuned to improve performance under the write-burst patterns produced by tools like Veeam Backup & Replication (tens of thousands of objects written in rapid succession):
  - **MemTable size: 64 MB** (up from the default 4 MB) — writes are buffered in RAM before being flushed to L0 SSTables. A larger MemTable absorbs longer bursts without triggering a flush, reducing write amplification and keeping the LSM tree cleaner.
  - **MemTableStopWritesThreshold: 12** — the number of unflushed MemTables before incoming writes are stalled. Raising this gives compaction more time to catch up during sustained write loads without blocking the client.
  - **L0StopWritesThreshold: 12** — similarly, writes are only stalled when 12 L0 files are pending compaction (default is 4). This prevents false write stalls during backup job peaks.
  - **Compaction concurrency: 2–4 goroutines** — Pebble now runs 2 to 4 background compaction goroutines simultaneously. This keeps the LSM tree compact between backup sessions, directly improving list and point-lookup latency.
  - **Bloom filters on L1–L6 (10 bits/key)** — a probabilistic structure that avoids unnecessary disk reads for point lookups where the key is absent. For example, a Veeam existence check (`HEAD /{bucket}/{object}`) on a key that doesn't exist would previously require scanning each LSM level on disk; with bloom filters the answer is ~99% certain in RAM. L0 deliberately has no bloom filter because range scans dominate there.
  - **Block size: 32 KB** — each SSTable block read from disk contains 32 KB of contiguous key-value data. Larger blocks are more efficient for sequential range scans (folder listings, prefix iteration) at the cost of slightly more memory per cached block.

  These values are fixed in the binary. The only user-tunable parameter is the cache size described above. (`internal/metadata/pebble_store.go`)

### Security
- **Bucket policy `Condition` blocks now evaluated** — `statementMatches` previously ignored the `Condition` field of every policy statement; conditions were silently skipped, meaning any policy that relied on `aws:SourceIp`, `aws:SecureTransport`, `s3:prefix`, or other condition keys had those restrictions completely unenforced. Implemented a full condition evaluator (`conditionMatches`) supporting the following operators: `StringEquals`, `StringNotEquals`, `StringEqualsIgnoreCase`, `StringNotEqualsIgnoreCase`, `StringLike`/`StringNotLike` (with AWS `*`/`?` wildcards), `Bool`, `IpAddress`/`NotIpAddress` (CIDR ranges via `net.ParseCIDR`), `ArnLike`/`ArnEquals`/`ArnNotEquals`/`ArnNotLike`, and `NumericEquals`/`NumericNotEquals`. The condition context is populated from the real HTTP request: `aws:SourceIp` maps to the client's IP (`net.SplitHostPort` of `r.RemoteAddr`), `aws:SecureTransport` maps to whether `r.TLS != nil`, and additional keys can be passed via a `RequestContext` map. All condition operators within a statement are ANDed together; within each operator all keys are also ANDed. Unknown operators fail closed (deny). `PolicyEvaluationRequest` gained three new fields: `SourceIP string`, `SecureTransport bool`, `RequestContext map[string]string`. (`internal/bucket/policy_evaluation.go`, `pkg/s3compat/handler.go`)
- **Bucket policy `Principal: null` granted access to everyone** — `principalMatches` returned `true` when `principal == nil`, treating a malformed or omitted Principal as a wildcard match. In AWS S3, `Principal` is a required field in bucket policy statements; a statement with a null/missing Principal is malformed and must not match any request. Any policy loaded from storage with a missing `Principal` field (e.g. a hand-edited or corrupted policy document) would silently allow any authenticated user through. Fixed by returning `false` for `nil` principal. (`internal/bucket/policy_evaluation.go`)
- **`GET /{bucket}/{key}?versions` removed — non-standard route returning 501** — this route was registered in `internal/api/handler.go` and mapped to a stub handler that unconditionally returned HTTP 501. The endpoint does not exist in the AWS S3 API; per-key version listing is accomplished via the standard `ListObjectVersions` (`GET /{bucket}?versions&prefix={key}`) which is already implemented. The dead route and its stub were removed. (`internal/api/handler.go`, `pkg/s3compat/handler.go`)

- **`InternalError` responses leaked internal filesystem paths** — every error path that called `h.writeError(w, "InternalError", err.Error(), ...)` forwarded the raw Go error string to the XML response body. On Windows, OS errors from `os.WriteFile` include the full absolute path (e.g. `C:\Users\aricardo\Projects\MaxIOFS\data\objects\...`). Similarly, storage errors in `StorageError.Error()` concatenate the OS cause. - **Presigned V2 URL and header-based V2 signatures always rejected** — Both the presigned URL validator (`calculateSignatureV2` in `presigned.go`) and the Authorization-header-based V2 verifier (`verifyS3SignatureV2` in `manager.go`) used `HMAC-SHA256` with hex encoding, but AWS S3 Signature Version 2 specifies `HMAC-SHA1` with standard base64 encoding (RFC 4648). Any presigned URL generated by the AWS CLI with `--signature-version s3` or any V2-capable SDK was rejected with "signature does not match". Fixed in both code paths by switching to `hmac.New(sha1.New, ...)` and `base64.StdEncoding.EncodeToString`. (`pkg/s3compat/presigned.go`, `internal/auth/manager.go`)

AWS S3 returns 404 for this error. Added to both the 404 status case and the `BucketName` resource field mapping. (`pkg/s3compat/handler.go`)
- **Auth error XML responses missing AWS context fields** — `InvalidAccessKeyId`, `SignatureDoesNotMatch`, and `RequestExpired` returned only `<Resource>` (the URL path) in the XML body. Added these fields to the `Error` struct and populated them in `writeError` by extracting the access key from `AWSAccessKeyId` query param (V2), `X-Amz-Credential` query param (V4), or `Authorization: AWS key:sig` header, and computing expiration times from `Expires` (V2) or `X-Amz-Date`+`X-Amz-Expires` (V4). ### Security
- **BUG-27 (MEDIUM): Open redirect via `success_action_redirect` in POST presigned uploads** — `HandlePostObject` redirected to any URL supplied in the `success_action_redirect` form field, including `javascript:` and `data:` schemes. Added a scheme whitelist check: only URLs starting with `http://` or `https://` are accepted; all other values return HTTP 400 `InvalidArgument`. (`pkg/s3compat/presigned.go`)
- **BUG-25 (CRITICAL): SSRF via webhook notification delivery** — `deliverNotification` used `http.DefaultClient` with no network restrictions, allowing an attacker with bucket-owner privileges to configure a notification webhook pointing to internal services (e.g. `http://169.254.169.254/`, `http://localhost:8081/`). Replaced with a custom `notifSSRFBlockingClient()` that dials through a `net.Dialer` which rejects loopback, link-local, and RFC-1918 private addresses before any HTTP bytes are sent. The client is injectable via `Handler.notifHTTPClient` so tests can bypass the SSRF check using a plain client. (`pkg/s3compat/notifications.go`, `pkg/s3compat/handler.go`)
- **BUG-26 (HIGH): No validation on notification webhook endpoint URL** — `PutBucketNotification` in `bucket_ops.go` accepted any string as a queue/topic/function ARN value without checking it was a valid HTTP URL. An attacker could store malformed or non-HTTP values that would later cause unhandled errors or unexpected behavior in the delivery goroutine. Added URL scheme validation: only `http://` and `https://` endpoints are accepted; others return `InvalidArgument`. (`pkg/s3compat/bucket_ops.go`)

### Added
- **S3: POST presigned URLs (HTML form upload)** — browsers can now upload files directly to a bucket via HTML `<form enctype="multipart/form-data">` using S3-compatible POST policy signatures. The server validates the policy expiration, HMAC-SHA256 signature (V4) or V2 signature, bucket/key/content-type conditions, starts-with prefix conditions, and content-length-range constraints before storing the object. Supports `success_action_redirect` (303), `success_action_status` (200/201/204), and `x-amz-meta-*` form fields. (`pkg/s3compat/presigned.go`, `internal/api/handler.go`)
- **S3: bucket notifications now stored and dispatched as webhooks** — `PutBucketNotification` and `GetBucketNotification` were previously no-ops; the configuration was accepted but discarded. Notification config is now persisted in the bucket metadata. After each mutating object operation (PutObject, DeleteObject, CopyObject, CompleteMultipartUpload), the bucket's notification rules are evaluated: event type is matched against configured patterns (supports trailing wildcard `s3:ObjectCreated:*`), object key is checked against prefix/suffix filters, and matching targets receive an S3-compatible JSON event payload via HTTP POST (non-blocking goroutine). Since MaxIOFS is not AWS, SNS/SQS/Lambda ARN values are treated as webhook HTTP endpoint URLs (same approach as MinIO). (`internal/metadata/types.go`, `internal/bucket/types.go`, `internal/bucket/interface.go`, `internal/bucket/manager_impl.go`, `internal/bucket/adapter.go`, `pkg/s3compat/bucket_ops.go`, `pkg/s3compat/notifications.go`, `pkg/s3compat/handler.go`, `pkg/s3compat/object_ops.go`, `pkg/s3compat/multipart.go`)
- **S3: per-bucket CORS rules are now applied to actual requests** — bucket CORS configurations stored via `PutBucketCORS` were saved to the metadata store but never enforced on incoming requests. Added `bucketCORSMiddleware` to the S3 API handler that looks up the bucket's CORS rules, matches the request `Origin` against the configured allowed origins and methods, and sets the appropriate `Access-Control-Allow-Origin`, `Access-Control-Allow-Methods`, `Access-Control-Allow-Headers`, `Access-Control-Expose-Headers`, and `Access-Control-Max-Age` headers. The middleware runs before `s3ClientMiddleware` so that browser `OPTIONS` preflight requests (which carry no auth headers) are answered correctly without being redirected to the web console. (`internal/api/handler.go`)
- **S3: lifecycle `Expiration.Days` / `Expiration.Date` rules now executed** — lifecycle rules with an `Expiration` action (delete objects older than N days, or older than a fixed date) were parsed and stored but the background lifecycle worker never evaluated them. Added `processObjectExpiration` to the worker; it lists all objects under the rule prefix, compares each object's `LastModified` against the configured cutoff (either `time.Now() - Days` or the literal `Date`), and calls `DeleteObject` for expired objects. On versioned buckets `DeleteObject` without a `versionId` creates a delete marker (correct S3 behavior). (`internal/lifecycle/worker.go`)
- **S3: lifecycle `AbortIncompleteMultipartUpload` rules now executed** — lifecycle rules with an `AbortIncompleteMultipartUpload` action were stored but never evaluated. Added `processAbortIncompleteMultipartUploads` to the worker; it lists all in-progress multipart uploads for the bucket, compares each upload's `Initiated` timestamp against `time.Now() - DaysAfterInitiation`, and calls `AbortMultipartUpload` for stale uploads. (`internal/lifecycle/worker.go`)
- **S3: multipart upload ETag now uses the correct `MD5(parts)-N` format** — `CompleteMultipartUpload` previously used only the MD5 of the concatenated part ETags as a hex string. The AWS S3 spec defines the multipart ETag as `hex(MD5(raw_binary_MD5_part1 || … || raw_binary_MD5_partN))-N`, where each part digest is the 16 raw bytes decoded from the hex ETag. Added `computeMultipartETag` to compute the spec-compliant value; the resulting ETag is stored in metadata and returned in the response. This enables clients such as `aws s3 sync --checksum` to correctly verify uploaded multipart objects. (`internal/object/manager.go`)

This caused the backup session to abort immediately after the object verification phase. Fixed in two layers: (1) `SetObjectRetention` in the object manager now accepts an optional `versionID ...string` parameter and uses `metadataStore.GetObject` with the specific version when provided; (2) the `PutObjectRetention` handler extracts the `versionId` query parameter and forwards it. Fixed: `SetObjectLockConfig` now sets `Versioning: Enabled` in the same `UpdateBucket` call whenever `ObjectLockEnabled` is true. (`internal/bucket/manager_impl.go`)

Root cause: `PutObjectVersion` stored a lightweight `ObjectVersion` struct (no `LegalHold`/`Retention` fields) at the versioned key; the full `ObjectMetadata` (with lock fields) was only at the non-versioned key. ## [1.0.0-rc1] - 2026-03-07

### Security
- **CRITICAL: AES-256-GCM replaces AES-256-CTR for object encryption** — AES-CTR provides confidentiality but no authentication: an attacker with write access to the storage backend could silently flip bits in ciphertext and the server would accept and decrypt the corrupted data (malleable ciphertext). Replaced with AES-256-GCM in 64 KB authenticated chunks. Each chunk has an independent nonce and authentication tag; any tampered chunk is detected and rejected on read. A backward-compatible legacy path decrypts existing AES-CTR objects transparently on first access; re-encryption happens lazily. New objects are always written with GCM.
- **CRITICAL: Cluster CA private key no longer transmitted over the network** — during cluster join, the CA private key was included in the HTTP response body, meaning any MITM or compromised node could steal it and sign arbitrary node certificates. Reworked to a CSR-based flow: the joining node generates its own ECDSA P-256 key pair locally, sends only a CSR to the `POST /sign-csr` endpoint, and the CA node signs it without ever transmitting the CA private key. The CA key now never leaves the originating node.
- **HIGH: 6 cluster management handlers missing `isGlobalAdmin` authorization check** — `handleAddClusterNode`, `handleRemoveClusterNode`, `handleInitializeCluster`, `handleJoinCluster`, `handleLeaveCluster`, and `handleGetClusterToken` accepted any authenticated request. Any tenant user or non-admin could initialize, join, or dismantle the cluster. All 6 handlers now enforce `isGlobalAdmin` before proceeding. Additionally, `handleAddClusterNode` had a partial SSRF: the caller-supplied node endpoint URL was fetched without validation. Added `validateClusterNodeEndpoint()` with SSRF-blocking dial restrictions.
- **HIGH: SSRF via notification webhook URLs** — the webhook notification system accepted arbitrary URLs from the database and made outbound HTTP requests without validation. An admin who set a webhook URL to `http://169.254.169.254/latest/meta-data/` (AWS metadata) or any internal service could exfiltrate instance credentials or probe the internal network. Added `ssrfBlockingDialer()` that blocks RFC 1918 private ranges, loopback, link-local (169.254.x.x), and IPv6 equivalents.
- **HIGH: SSRF via external HTTP logging targets** — same pattern as webhooks: user-configured HTTP log destinations were fetched without SSRF protection. Applied the same `ssrfBlockingDialer()` to the HTTP logging output.
- **HIGH: Path traversal via backslash on Windows** — `filesystem.go` sanitized `/` separators but not `\`, allowing object keys like `..\..\etc\passwd` to escape the data directory on Windows hosts. Added explicit `\` blocking, post-join canonical path validation with `filepath.Clean`, and `validatePath` in `List()`.
- **HIGH: Replication credentials stored in plaintext** — target bucket secret keys were stored unencrypted in the replication SQLite database. Any user with read access to the database file could extract valid AWS-compatible credentials. Now encrypted with AES-256-GCM using the server's `SecretKey`; stored with an `enc1:` prefix. Existing plaintext entries are read and re-encrypted on first access.
- **HIGH: HMAC nonce derived from `time.Now().UnixNano()` (predictable)** — inter-node cluster authentication used `time.Now().UnixNano()` as a nonce, making it predictable within a known time window and potentially replayable. Replaced with 16 bytes from `crypto/rand`.
- **HIGH: Infinite cluster proxy loop** — when node A proxied a request to node B, B could proxy it back to A (A→B→A→…) if routing logic disagreed. Added `X-MaxIOFS-Proxied: true` header; nodes reject any incoming request already carrying this header with 508 Loop Detected.
- **MEDIUM: OAuth callback missing CSRF state validation** — the Google OAuth callback accepted any `code` parameter without verifying the `state` value against the session, making it vulnerable to CSRF login attacks. Added `state` parameter generation, storage, and strict validation on callback.
- **MEDIUM: CORS origin allowlist replaced wildcard** — `Access-Control-Allow-Origin: *` was set unconditionally on all responses, allowing any website to make credentialed cross-origin requests to the console API. Replaced with `CORSWithConfig` using an explicit origin allowlist derived from `server.public_console_url`.
- **MEDIUM: DoS via multipart upload without part count limit** — a client could open a multipart upload and submit an unlimited number of parts (each consuming server memory and disk), eventually exhausting resources. Added a 10,000-part cap per upload (matching the S3 spec) and `partNumber` validation (must be 1–10,000).
- **MEDIUM: SSRF via replication destination endpoint** — user-configured replication target URLs were passed directly to the AWS SDK HTTP client without validation. Added `validateReplicationEndpoint()` and `ssrfBlockingReplicationDialer()` injected into the SDK's transport.
- **Frontend MEDIUM: Auth cookies missing `Secure` and `SameSite` flags** — JWT auth cookies were set without `Secure` (transmittable over plain HTTP) or `SameSite=Strict` (susceptible to CSRF). Added both flags to all `document.cookie` write and clear operations in `api.ts` and `oauth-complete.tsx`.
- **Frontend MEDIUM: Incomplete HTML sanitizer** — `sanitizeHtml()` in `modals.tsx` only blocked lowercase `javascript:`; missed `JAVASCRIPT:`, `data:` URIs, whitespace padding (`javascript :`) and HTML-entity variants. Rewrote sanitizer using the browser's own HTML parser for entity decoding, strict URL attribute inspection via `getAttribute()`, and an `http/https/mailto/#/` allowlist. Also removes `<meta>` and `<base>` tags.
- **Frontend MEDIUM: XSS via server values interpolated into HTML templates** — `shareData.url`, object keys, and bucket names returned by the server were interpolated directly into HTML template literals passed as `html:` to `ModalManager.fire()`. If the server were compromised or values contained HTML metacharacters, this would execute arbitrary scripts. Added `escapeHtml()` to `src/lib/utils.ts` and applied at every dynamic interpolation site.
- **Frontend LOW: Debug `console.log` statements in production** — `settings.tsx` and `users/[user]/index.tsx` left debug log calls that exposed ACL/policy API responses and internal user IDs in the browser DevTools console. Removed all non-error debug statements.

### Changed
- **Frontend: code splitting with `React.lazy`** — 18 pages now load on demand; main bundle reduced from **1,003 kB → 550 kB** (gzip: 252 kB → 170 kB, −33%). The recharts library (383 kB) only loads when navigating to the Metrics page.

- **Debian/RPM packages: service auto-restart on upgrade** — `prerm` now records whether the service was running (flag file `/tmp/maxiofs-service-was-running`) before stopping it; `postinst` restarts it after upgrade if the flag is present. Services manually stopped before upgrading are not auto-started. RPM `%post` runs `try-restart` when `$1 -gt 1`.
- **Makefile: `VERSION_CLEAN` strips `-rc*` suffix** — `v1.0.0-rc1` now produces `1.0.0` for RPM/tarball naming, matching the existing `-beta`/`-alpha` stripping behaviour.

### Fixed
- **Mobile sidebar stays open after navigation** — clicking a nav link on small screens navigated to the page but left the sidebar overlay open; the user had to tap outside to dismiss it. All nav `<Link>` elements in `SidebarNav` now call `onClose()` on click.

### Tests

### Fixed
- **Empty folders not displayed in object browser** — folder marker objects (keys ending in `/`, stored with metadata flag `x-maxiofs-implicit-folder=true`) were unconditionally skipped in `ListObjects` and `SearchObjects`. When a folder contained no objects, there were no child-key paths to derive the common prefix from, so the folder was completely invisible in the frontend. The fix: when a delimiter is present (hierarchical listing mode), folder markers are no longer skipped — they fall through to the common-prefix extraction block and are added to `commonPrefixesMap` like any other object whose key contains the delimiter after the prefix. Flat listings (`delimiter=""`) continue to omit implicit folder markers as before. A guard prevents the folder's own key from appearing as a child of itself when listing with the folder as the prefix.
- **Integrity scanner erroring on folder and delete-marker objects** — `VerifyBucketIntegrity` calls `metadataStore.ListObjects` directly (bypassing the object-manager filters) and therefore received folder markers and delete markers alongside real data objects. `VerifyObjectIntegrity` would then try to open a content file for a folder key (which has no backing file on storage) and report it as `IntegrityMissing` or `IntegrityError`. Fixed by adding two early-exit checks at the top of `VerifyObjectIntegrity`: (1) any key ending with `/` is returned as `IntegritySkipped` with reason `"folder marker: not a data object"`; (2) any object whose stored ETag is empty (delete marker tombstone) is returned as `IntegritySkipped` with reason `"delete marker: not a data object"`. Both types are counted in the `skipped` total so the report counters remain consistent (`checked = ok + corrupted + skipped + errors`).
- **Integrity scan results persisted and surfaced in UI** — previously, opening the bucket integrity modal always showed "Ready to scan" regardless of whether the background scrubber had already run. Scan results are now persisted per-bucket in Pebble (`integrity_scans:{bucketPath}` key) as a JSON array of the last **10** scan records (newest first). The scrubber accumulates per-bucket totals across all paginated pages and writes one record on completion. On-demand manual scans post their accumulated result via `POST /buckets/{bucket}/integrity-status` when finished. The modal fetches the history via `GET /buckets/{bucket}/integrity-status` on open and immediately shows the last known state (including which source triggered it — `"manual"` or `"scrubber"`). Old single-record format is read with backward-compat fallback.
- **Integrity check modal permissions** — "Start Verification", "Scan again" and "Retry" action buttons are hidden for tenant admins (view-only access to scan history). Regular users no longer see the "Check Integrity" button in the bucket list at all, preventing unnecessary 403 requests to the backend. New `canRunScan` prop on `BucketIntegrityModal` (default `false`); passed as `canRunScan={isGlobalAdmin}` from the bucket list.
- **Integrity scan rate limit** — a global admin could repeatedly trigger `POST /buckets/{bucket}/verify-integrity` back-to-back, causing unbounded disk I/O. Manual scans are now rate-limited to **once per hour** per bucket. The limit is enforced server-side: when `marker` is empty (start of a new scan), the handler checks the timestamp of the most recent manual scan from the persisted history. If less than 60 minutes have elapsed, it returns `HTTP 429` with a message indicating the remaining wait time. Continuation pages (`marker != ""`) are always allowed. The frontend surfaces the limit proactively: the modal computes the next-available time from history and shows a live countdown with the "Start Verification" / "Scan again" button disabled until the cooldown expires.
- **SMTP TLS mode ambiguity** — `email.use_tls: false` (previous default) connected to the SMTP server and then opportunistically upgraded to STARTTLS whenever the server advertised it, making it impossible to force a truly unencrypted connection (e.g. internal Postfix relay or Milter setup that does not support TLS). The setting has been replaced with an explicit three-way `email.tls_mode` selector: `none` (plain TCP, no TLS at all — default), `starttls` (connect plain, upgrade with STARTTLS; fails if server does not advertise it), `ssl` (implicit TLS from the first byte, port 465). An additional `email.skip_tls_verify` boolean allows bypassing certificate validation for self-signed certificates on internal mail servers. Existing `email.use_tls` entries are removed from the database on startup as a deprecated setting.
- **Virtual-hosted-style S3 requests returning bucket list instead of objects** — when a client used virtual-hosted-style addressing (bucket name in the subdomain, e.g. `mybucket.s3.example.com`), every request to `/` or any object path was incorrectly routed to the root handler and returned a `ListAllMyBucketsResult` instead of the expected bucket operation. This broke compatibility with WinSCP, CyberDuck, CloudBerry, and other clients that default to virtual-hosted-style. Fixed by adding an HTTP middleware (`virtualHostedStyleMiddleware`) that intercepts requests before Gorilla Mux routing: if the `Host` header matches `{bucket}.{configured-s3-host}`, the URL path is transparently rewritten from `/...` to `/{bucket}/...` so all existing path-based routes handle it correctly. Both path-style and virtual-hosted-style now work simultaneously without any configuration change.
- **Metadata corruption on unclean shutdown** — Pebble's WAL guarantees database integrity after process kill, power loss, or OOM kill. No more `MANIFEST` corruption requiring manual recovery.
- **Object/size metrics corrected post-migration** — the BadgerDB → Pebble migration recounted all existing objects and sizes, fixing counters that had diverged due to BadgerDB transaction conflicts under concurrent load.
- **Bucket metrics under-reported under concurrent load (VEEAM / multiple S3 clients)** — `UpdateBucketMetrics` used BadgerDB's Optimistic Concurrency Control (OCC) with only 5 retry attempts. Under high concurrency (VEEAM Backup with multiple parallel upload threads, 10 or 100 simultaneous S3 clients), `ErrConflict` exhausted retries and silently discarded metric updates, causing the web interface to show significantly less storage than was actually stored. The retry loop has been replaced with a per-bucket `sync.Mutex` via `sync.Map`: serialization now occurs at the Go level, making `ErrConflict` impossible by construction. Metric updates are now fully reliable under any concurrency level.
- **`RecalculateBucketStats` ignored tenant prefix** — the function scanned `obj:bucketName:` but objects in tenant buckets are stored as `obj:tenantID/bucketName:key`. For any tenant bucket, recalculation always returned 0 objects and 0 bytes, silently resetting counters. Now builds the full path (`tenantID/bucketName` for tenant buckets, or just `bucketName` for global buckets) before scanning. Global buckets (no tenant) are unaffected.

- **Large file multipart upload — 5 cascading bugs fixed** (`f10b774`, resolves [#5](https://github.com/MaxIOFS/MaxIOFS/issues/5)):
  1. *Error type mismatch*: HTTP handler checked `ErrUploadNotFound` but the object manager returns `ErrInvalidUploadID`; fell through to a generic 500 instead of 404.
  2. *`os.Rename` race on Windows*: two concurrent `CompleteMultipartUpload` requests for the same `uploadID` raced to rename the temp file to `objectPath`. On Windows, `os.Open` does not set `FILE_SHARE_DELETE`, causing `ERROR_SHARING_VIOLATION`. Fixed with per-`uploadID` deduplication (`completionFuture`).
  3. *Excessive I/O (3× full-file read/write)*: `calculateMultipartHash` re-read the entire combined object to compute an MD5 already computed by `storage.Put`; `storeUnencryptedMultipartObject` then re-read and re-wrote the same file just to update the `.metadata` sidecar. Metadata operations then failed, leaving multipart state partially cleaned up; retries got `NoSuchUpload`. Fixed: goroutine now uses `context.WithoutCancel(r.Context())`.
